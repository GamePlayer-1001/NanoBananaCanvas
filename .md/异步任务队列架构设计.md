# Nano Banana Canvas - 异步任务队列架构设计

> 文档版本：v2.0
> 创建日期：2026-02-26
> 最后更新：2026-03-10
>
> **[STATUS: P2 开发中 — Method C 架构]**
> 本文档描述基于 D1-as-Queue + 服务端代理 + 客户端驱动轮询的异步任务系统。
> Method C 的核心理念：**前端 poll 就是调度器**——客户端轮询 GET /api/tasks/:id 时，
> API Route 按需查询外部 Provider 状态（懒评估），D1 既是状态存储也是任务队列。
> 无 Cloudflare Queues 依赖，无独立 Consumer Worker，无额外付费服务。

---

## 一、概述

### 1.1 设计目标

Nano Banana Canvas 作为 AI 工作流编排平台，核心任务之一是处理耗时较长的 AI 生成任务。不同类型的任务有显著不同的响应时间特征：

| 任务类型       | 典型耗时  | 示例                                  |
| -------------- | --------- | ------------------------------------- |
| 视频生成       | 2-10 分钟 | Kling 视频生成、Wan 文生视频/首帧尾帧 |
| 复杂图片生成   | 30s-2min  | QWEN 图片生成、Z-Image、高分辨率 FLUX |
| 音频生成 (TTS) | 10-60s    | 长文本语音合成                        |
| 简单图片生成   | 5-30s     | 低分辨率 SD/FLUX                      |

这些任务的共同特征：**无法在单次 HTTP 请求中完成**。因此需要一套完整的异步任务处理系统。

### 1.2 为什么需要异步

**技术约束**：

- Cloudflare Workers CPU 时间限制：Free 计划 10ms，Paid 计划 30s（非 CPU 等待不计入）
- 即使使用 Paid 计划，30s CPU 时间不足以覆盖视频生成任务的完整生命周期
- 外部 AI API（Kling、Wan 等）本身就是异步设计——提交任务后需轮询获取结果

**用户体验**：

- 长时间等待同步响应会导致浏览器超时、连接中断
- 用户应能在提交任务后继续编辑其他节点
- 任务进度需实时可见（进度条、百分比）

**系统可靠性**：

- 异步解耦允许独立的重试机制
- 失败任务不阻塞后续任务
- 支持优雅降级（高峰期排队而非拒绝）

### 1.3 核心架构图

```
                    ┌─────────────┐
                    │  Next.js    │
                    │  Frontend   │
                    └──────┬──────┘
                           │ POST /api/tasks (submit)
                           │ GET  /api/tasks/:id (poll + lazy check)
                           │ POST /api/tasks/:id/cancel
                           ▼
                    ┌─────────────┐     ┌───────────┐
                    │  Next.js    │────▶│ External  │
                    │  API Routes │◀────│ AI APIs   │
                    └──────┬──────┘     └───────────┘
                           │
                    ┌──────┴──────┐
                    ▼             ▼
              ┌──────────┐ ┌──────────┐
              │    D1    │ │    R2    │
              │(Queue+   │ │(生成文件) │
              │ State)   │ │          │
              └──────────┘ └──────────┘
```

**Method C 核心思想**：

- **D1 即队列**：`async_tasks` 表同时承担任务队列和状态存储的双重角色
- **API Route 即调度器**：无独立 Consumer Worker，所有逻辑在 Next.js API Routes 中执行
- **客户端即触发器**：前端轮询 GET 请求触发 Provider 状态检查（懒评估）
- **用户自带 Key**：平台不代理 AI API 服务，用户在项目中配置自己的 Provider API Key

## 二、任务生命周期

### 2.1 状态机定义

```
    ┌─────────┐     ┌──────────┐
    │ pending │────▶│ running  │
    └─────────┘     └────┬─────┘
                         │
               ┌─────────┼─────────┐
               ▼         ▼         ▼
        ┌───────────┐ ┌────────┐ ┌───────────┐
        │ completed │ │ failed │ │ cancelled │
        └───────────┘ └───┬────┘ └───────────┘
                          │
                    (retry_count < max_retries)
                          │
                    ┌─────┴─────┐
                    │  重新提交  │──▶ running
                    └───────────┘
```

五个状态，三个终态，零中间态。简化是最高形式的复杂。

### 2.2 状态定义

| 状态        | 描述                       | 存储位置 | 进入条件                                   |
| ----------- | -------------------------- | -------- | ------------------------------------------ |
| `pending`   | 已验证，Provider 尚未返回  | D1       | 创建任务，准备调用 Provider                |
| `running`   | Provider 已接受，正在处理  | D1       | Provider.submit() 成功返回 externalTaskId  |
| `completed` | 任务成功，结果已存储       | D1 + R2  | Provider 返回完成 + 结果 URL               |
| `failed`    | 重试耗尽或不可恢复错误     | D1       | retry_count >= max_retries 或致命错误      |
| `cancelled` | 用户主动取消               | D1       | 用户调用 cancel API                        |

### 2.3 状态转换规则

```typescript
// ============================================================
//  任务状态转换规则 — Method C 五态模型
// ============================================================

type TaskStatus = 'pending' | 'running' | 'completed' | 'failed' | 'cancelled'

const VALID_TRANSITIONS: Record<TaskStatus, TaskStatus[]> = {
  pending:   ['running', 'failed', 'cancelled'],  // submit 成功 / submit 失败 / 用户取消
  running:   ['completed', 'failed', 'cancelled'], // Provider 完成 / Provider 失败 / 用户取消
  completed: [],                                    // 终态
  failed:    ['running'],                           // 重试: 重新 submit → running
  cancelled: [],                                    // 终态
}

// ----------------------------------------------------------
//  终态判断 — 轮询停止条件
// ----------------------------------------------------------

const TERMINAL_STATUSES: Set<TaskStatus> = new Set(['completed', 'failed', 'cancelled'])

function isTerminal(status: TaskStatus): boolean {
  return TERMINAL_STATUSES.has(status)
}
```

### 2.4 超时策略

| 任务类型 | 全局超时（从创建到超时） | 轮询间隔（前端） | 最大重试次数 |
| -------- | ------------------------ | ----------------- | ------------ |
| 视频生成 | 15 分钟                  | 5 秒              | 2            |
| 图片生成 | 15 分钟                  | 5 秒              | 2            |
| 音频生成 | 3 分钟                   | 3 秒              | 2            |

```typescript
// ============================================================
//  任务配置 — 按任务类型定义超时、轮询间隔、重试上限
// ============================================================

type AsyncTaskType = 'video_gen' | 'image_gen' | 'audio_gen'

interface TaskTypeConfig {
  timeoutMs: number       // 全局超时 (ms), 从 created_at 起算
  pollIntervalMs: number  // 前端轮询间隔 (ms)
  maxRetries: number      // 最大重试次数
  throttleMs: number      // checkStatus 节流间隔 (ms), 防止短时间内重复查询 Provider
}

const TASK_CONFIG: Record<AsyncTaskType, TaskTypeConfig> = {
  video_gen: {
    timeoutMs: 15 * 60 * 1000,     // 15 分钟
    pollIntervalMs: 5000,           // 5 秒
    maxRetries: 2,
    throttleMs: 10_000,             // 10 秒内不重复查询 Provider
  },
  image_gen: {
    timeoutMs: 15 * 60 * 1000,     // 15 分钟
    pollIntervalMs: 5000,           // 5 秒
    maxRetries: 2,
    throttleMs: 10_000,             // 10 秒
  },
  audio_gen: {
    timeoutMs: 3 * 60 * 1000,      // 3 分钟
    pollIntervalMs: 3000,           // 3 秒
    maxRetries: 2,
    throttleMs: 5_000,              // 5 秒
  },
}
```

## 三、D1-as-Queue + 客户端驱动轮询

### 3.1 架构决策与权衡

**为什么选择 Method C？**

| 维度           | Cloudflare Queues         | Cron Trigger            | Method C (D1-as-Queue)     |
| -------------- | ------------------------- | ----------------------- | -------------------------- |
| 成本           | $0.40/百万消息 (付费)     | 免费（但浪费调用）      | 免费                       |
| 架构复杂度     | 高（Producer + Consumer） | 中（Cron + Worker）     | 低（API Routes 即一切）    |
| 独立 Worker    | 需要                      | 需要                    | 不需要                     |
| 实时性         | 近实时                    | 取决于 Cron 频率        | 取决于前端轮询频率         |
| 空闲时资源消耗 | 无                        | 有（Cron 空转）         | 无（懒评估）               |
| 适用规模       | 大规模 (10k+/天)          | 中等规模                | 中小规模 (<1k/天)          |
| 调试难度       | 高（异步消息难追踪）      | 中                      | 低（请求 → 响应，线性流）  |

**Method C 核心洞察**：

> 外部 AI Provider（Kling、Wan、QWEN）本身就是异步的——你提交任务后轮询它们的状态。
> 既然前端已经在轮询我们的 API，我们的 API 在被轮询时去查 Provider 状态，这就是最自然的调度方式。
> 不需要一个独立的 Consumer Worker 持续轮询——**让用户的请求驱动一切**。

**唯一权衡**：Provider 结果有过期时间（通常 24h），如果用户提交任务后长时间不回来轮询，结果可能丢失。后期可加 Cron 兜底扫描 running 状态的任务，但 MVP 阶段这个风险极低。

### 3.2 D1 任务表设计

```sql
-- ============================================================
--  async_tasks — D1 即队列，即状态存储
--  Method C 的唯一数据结构，承载任务全生命周期
-- ============================================================

CREATE TABLE IF NOT EXISTS async_tasks (
  -- 主键
  id                TEXT PRIMARY KEY,              -- nanoid 生成

  -- 关联关系
  user_id           TEXT NOT NULL,                 -- Clerk user ID
  project_id        TEXT,                          -- 所属项目 ID (可选)
  workflow_id       TEXT,                          -- 所属工作流 ID (可选)
  node_id           TEXT,                          -- 所属节点 ID (可选)

  -- 任务定义
  type              TEXT NOT NULL,                 -- 'video_gen' | 'image_gen' | 'audio_gen'
  provider          TEXT NOT NULL,                 -- 'kling' | 'wan' | 'qwen' | 'z-image'
  model_id          TEXT NOT NULL,                 -- 'kling-v1' | 'wan-text2video' 等

  -- 状态跟踪 — 五态模型
  status            TEXT NOT NULL DEFAULT 'pending',  -- pending/running/completed/failed/cancelled
  progress          INTEGER NOT NULL DEFAULT 0,       -- 0-100

  -- 数据载荷
  input_data        TEXT NOT NULL,                 -- JSON: 输入参数 (prompt, settings 等)
  output_data       TEXT,                          -- JSON: 输出结果 (文件 URL 列表等)

  -- 外部 Provider 关联
  external_task_id  TEXT,                          -- Provider 返回的任务 ID

  -- 积分 — 三阶段模型
  credits_charged   INTEGER NOT NULL DEFAULT 0,    -- 冻结的积分数 (submit 时 freeze)

  -- 重试
  retry_count       INTEGER NOT NULL DEFAULT 0,    -- 已重试次数
  error_message     TEXT,                          -- 最近的错误信息

  -- Provider 状态检查节流
  last_checked_at   INTEGER,                       -- 上次查询 Provider 的时间戳 (ms)

  -- 时间戳
  created_at        INTEGER NOT NULL,              -- 创建时间 (Unix ms)
  started_at        INTEGER,                       -- Provider 接受时间
  completed_at      INTEGER                        -- 终态时间 (完成/失败/取消)
);

-- ============================================================
--  索引设计 — 针对 Method C 查询模式优化
-- ============================================================

-- 用户任务列表（最常用：任务面板）
CREATE INDEX idx_tasks_user_status ON async_tasks(user_id, status);

-- 并发控制：统计用户活跃任务数
CREATE INDEX idx_tasks_user_active ON async_tasks(user_id, status)
  WHERE status IN ('pending', 'running');

-- 按创建时间排序（分页查询用）
CREATE INDEX idx_tasks_user_created ON async_tasks(user_id, created_at DESC);

-- 超时检测：找出过期的 running/pending 任务
CREATE INDEX idx_tasks_timeout ON async_tasks(status, created_at)
  WHERE status IN ('pending', 'running');

-- 工作流关联查询
CREATE INDEX idx_tasks_workflow ON async_tasks(workflow_id)
  WHERE workflow_id IS NOT NULL;
```

### 3.3 提交流程（submitTask）

```typescript
// ============================================================
//  POST /api/tasks — 提交异步任务
//  app/api/tasks/route.ts
//
//  流程: auth → 并发检查 → freeze credits → processor.submit → INSERT D1
// ============================================================

import { nanoid } from 'nanoid'

export async function POST(request: Request) {
  const { userId } = auth()
  if (!userId) return Response.json({ error: 'Unauthorized' }, { status: 401 })

  const body = await request.json()
  const { type, provider, modelId, input, projectId, workflowId, nodeId } = body

  // ----------------------------------------------------------
  // Step 1: 获取用户的 Provider API Key
  // ----------------------------------------------------------
  const apiKey = await getUserProviderKey(db, userId, projectId, provider)
  if (!apiKey) {
    return Response.json(
      {
        error: 'MISSING_API_KEY',
        message: `请先在项目设置中配置 ${provider} 的 API Key`,
      },
      { status: 400 },
    )
  }

  // ----------------------------------------------------------
  // Step 2: 并发限制检查
  // ----------------------------------------------------------
  const concurrency = await checkConcurrency(db, userId)
  if (!concurrency.allowed) {
    return Response.json(
      {
        error: 'CONCURRENT_LIMIT_EXCEEDED',
        message: concurrency.reason,
        currentCount: concurrency.currentCount,
        maxCount: concurrency.maxCount,
      },
      { status: 429 },
    )
  }

  // ----------------------------------------------------------
  // Step 3: 冻结积分（freeze，非最终扣减）
  // ----------------------------------------------------------
  const creditsRequired = getModelCredits(provider, modelId)
  const freezeResult = await freezeCredits(db, userId, creditsRequired)
  if (!freezeResult.success) {
    return Response.json(
      {
        error: 'INSUFFICIENT_CREDITS',
        message: `积分不足，需要 ${creditsRequired}，当前余额 ${freezeResult.balance}`,
      },
      { status: 402 },
    )
  }

  // ----------------------------------------------------------
  // Step 4: 获取 Processor 并提交到外部 Provider
  // ----------------------------------------------------------
  const taskId = nanoid()
  const now = Date.now()
  const processor = ProcessorRegistry.get(type)

  let externalTaskId: string | null = null
  let initialStatus: 'pending' | 'running' = 'pending'

  try {
    const submitResult = await processor.submit(
      { provider, modelId, input, taskId, userId },
      apiKey,
    )
    externalTaskId = submitResult.externalTaskId
    initialStatus = 'running'
  } catch (error) {
    // Provider submit 失败 — 仍创建任务记录，标记为 pending
    // 后续 checkTask 时会检测到并进行重试
    console.error(`Provider submit failed for task ${taskId}:`, error)
  }

  // ----------------------------------------------------------
  // Step 5: INSERT 任务记录到 D1
  // ----------------------------------------------------------
  await db
    .prepare(
      `
      INSERT INTO async_tasks (
        id, user_id, project_id, workflow_id, node_id,
        type, provider, model_id, status,
        input_data, external_task_id, credits_charged,
        created_at, started_at
      ) VALUES (?, ?, ?, ?, ?, ?, ?, ?, ?, ?, ?, ?, ?, ?)
    `,
    )
    .bind(
      taskId,
      userId,
      projectId ?? null,
      workflowId ?? null,
      nodeId ?? null,
      type,
      provider,
      modelId,
      initialStatus,
      JSON.stringify(input),
      externalTaskId,
      creditsRequired,
      now,
      externalTaskId ? now : null,
    )
    .run()

  return Response.json({ taskId, status: initialStatus }, { status: 201 })
}
```

### 3.4 懒评估轮询（checkTask）

这是 Method C 的核心——前端轮询 GET 请求触发 Provider 状态检查。

```typescript
// ============================================================
//  GET /api/tasks/:id — 懒评估轮询
//  app/api/tasks/[id]/route.ts
//
//  流程: SELECT D1 → 终态直接返回
//        非终态 → 节流检查 → processor.checkStatus
//        → UPDATE D1 → confirm/refund/retry → 超时检测
// ============================================================

export async function GET(
  request: Request,
  { params }: { params: { id: string } },
) {
  const { userId } = auth()
  if (!userId) return Response.json({ error: 'Unauthorized' }, { status: 401 })

  // ----------------------------------------------------------
  // Step 1: 从 D1 读取任务当前状态
  // ----------------------------------------------------------
  const task = await db
    .prepare('SELECT * FROM async_tasks WHERE id = ? AND user_id = ?')
    .bind(params.id, userId)
    .first()

  if (!task) {
    return Response.json({ error: 'Task not found' }, { status: 404 })
  }

  const status = task.status as string
  const taskType = task.type as AsyncTaskType

  // ----------------------------------------------------------
  // Step 2: 终态 — 直接返回，不查 Provider
  // ----------------------------------------------------------
  if (isTerminal(status as TaskStatus)) {
    return Response.json(formatTaskResponse(task))
  }

  // ----------------------------------------------------------
  // Step 3: 超时检测 — created_at + timeoutMs < now
  // ----------------------------------------------------------
  const now = Date.now()
  const config = TASK_CONFIG[taskType]
  const createdAt = task.created_at as number

  if (now - createdAt > config.timeoutMs) {
    // 超时 → 标记 failed + 退还积分
    await db
      .prepare(
        `UPDATE async_tasks
         SET status = 'failed', error_message = '任务执行超时', completed_at = ?
         WHERE id = ?`,
      )
      .bind(now, task.id)
      .run()

    await refundCredits(db, userId, task.credits_charged as number, task.id as string)

    return Response.json(
      formatTaskResponse({
        ...task,
        status: 'failed',
        error_message: '任务执行超时',
        completed_at: now,
      }),
    )
  }

  // ----------------------------------------------------------
  // Step 4: pending 且无 externalTaskId — 需要重新 submit
  // ----------------------------------------------------------
  if (status === 'pending' && !task.external_task_id) {
    const apiKey = await getUserProviderKey(
      db, userId, task.project_id as string, task.provider as string,
    )
    if (!apiKey) {
      return Response.json(formatTaskResponse(task))
    }

    try {
      const processor = ProcessorRegistry.get(taskType)
      const submitResult = await processor.submit(
        {
          provider: task.provider as string,
          modelId: task.model_id as string,
          input: JSON.parse(task.input_data as string),
          taskId: task.id as string,
          userId,
        },
        apiKey,
      )

      await db
        .prepare(
          `UPDATE async_tasks
           SET status = 'running', external_task_id = ?, started_at = ?, last_checked_at = ?
           WHERE id = ?`,
        )
        .bind(submitResult.externalTaskId, now, now, task.id)
        .run()

      return Response.json(
        formatTaskResponse({
          ...task,
          status: 'running',
          external_task_id: submitResult.externalTaskId,
          started_at: now,
        }),
      )
    } catch {
      // submit 再次失败，保持 pending，等下次轮询
      return Response.json(formatTaskResponse(task))
    }
  }

  // ----------------------------------------------------------
  // Step 5: running — 节流检查 Provider 状态
  // ----------------------------------------------------------
  if (status === 'running' && task.external_task_id) {
    const lastChecked = (task.last_checked_at as number) ?? 0
    const throttleMs = config.throttleMs

    // 节流: 距离上次检查不足 throttleMs，直接返回缓存状态
    if (now - lastChecked < throttleMs) {
      return Response.json(formatTaskResponse(task))
    }

    const apiKey = await getUserProviderKey(
      db, userId, task.project_id as string, task.provider as string,
    )
    if (!apiKey) {
      return Response.json(formatTaskResponse(task))
    }

    try {
      const processor = ProcessorRegistry.get(taskType)
      const checkResult = await processor.checkStatus(
        task.external_task_id as string,
        apiKey,
      )

      // 更新 last_checked_at（无论结果如何）
      const updates: Record<string, unknown> = { last_checked_at: now }

      if (checkResult.progress !== undefined) {
        updates.progress = checkResult.progress
      }

      if (checkResult.status === 'completed' && checkResult.output) {
        // ------ 任务完成 ------
        updates.status = 'completed'
        updates.output_data = JSON.stringify(checkResult.output)
        updates.progress = 100
        updates.completed_at = now

        await updateTask(db, task.id as string, updates)
        await confirmCredits(db, userId, task.credits_charged as number, task.id as string)

        return Response.json(
          formatTaskResponse({ ...task, ...updates }),
        )
      }

      if (checkResult.status === 'failed') {
        // ------ Provider 报告失败 ------
        const retryCount = (task.retry_count as number) ?? 0
        const isRetryable = checkResult.retryable !== false
        const canRetry = isRetryable && retryCount < config.maxRetries

        if (canRetry) {
          // 重新提交
          try {
            const resubmitResult = await processor.submit(
              {
                provider: task.provider as string,
                modelId: task.model_id as string,
                input: JSON.parse(task.input_data as string),
                taskId: task.id as string,
                userId,
              },
              apiKey,
            )
            updates.external_task_id = resubmitResult.externalTaskId
            updates.retry_count = retryCount + 1
            updates.status = 'running'
            updates.error_message = checkResult.error
            updates.started_at = now
          } catch {
            // 重新提交也失败了
            updates.retry_count = retryCount + 1
            updates.error_message = checkResult.error
          }
        } else {
          // 不可重试或重试耗尽 → 最终失败
          updates.status = 'failed'
          updates.error_message = checkResult.error ?? '任务处理失败'
          updates.completed_at = now

          await updateTask(db, task.id as string, updates)
          await refundCredits(db, userId, task.credits_charged as number, task.id as string)

          return Response.json(
            formatTaskResponse({ ...task, ...updates }),
          )
        }
      }

      // ------ 仍在运行 ------
      await updateTask(db, task.id as string, updates)
      return Response.json(
        formatTaskResponse({ ...task, ...updates }),
      )
    } catch (error) {
      // Provider 查询本身失败（网络错误等），不改变任务状态
      console.error(`Provider check failed for task ${task.id}:`, error)
      await db
        .prepare('UPDATE async_tasks SET last_checked_at = ? WHERE id = ?')
        .bind(now, task.id)
        .run()
      return Response.json(formatTaskResponse(task))
    }
  }

  // 兜底返回
  return Response.json(formatTaskResponse(task))
}

// ----------------------------------------------------------
//  辅助: 批量更新任务字段
// ----------------------------------------------------------
async function updateTask(
  db: D1Database,
  taskId: string,
  updates: Record<string, unknown>,
): Promise<void> {
  const keys = Object.keys(updates)
  if (keys.length === 0) return

  const setClauses = keys.map((k) => `${k} = ?`).join(', ')
  const values = keys.map((k) => updates[k])

  await db
    .prepare(`UPDATE async_tasks SET ${setClauses} WHERE id = ?`)
    .bind(...values, taskId)
    .run()
}
```

### 3.5 取消流程（cancelTask）

```typescript
// ============================================================
//  POST /api/tasks/:id/cancel — 取消任务
//  app/api/tasks/[id]/cancel/route.ts
//
//  流程: processor.cancel → UPDATE status=cancelled → refundCredits
// ============================================================

export async function POST(
  request: Request,
  { params }: { params: { id: string } },
) {
  const { userId } = auth()
  if (!userId) return Response.json({ error: 'Unauthorized' }, { status: 401 })

  const task = await db
    .prepare('SELECT * FROM async_tasks WHERE id = ? AND user_id = ?')
    .bind(params.id, userId)
    .first()

  if (!task) {
    return Response.json({ error: 'Task not found' }, { status: 404 })
  }

  const status = task.status as string

  // 只有 pending 和 running 状态可以取消
  if (!['pending', 'running'].includes(status)) {
    return Response.json(
      {
        success: false,
        message: `无法取消 ${status} 状态的任务`,
      },
      { status: 400 },
    )
  }

  // ----------------------------------------------------------
  // 尝试通知 Provider 取消（best effort，不阻塞）
  // ----------------------------------------------------------
  if (task.external_task_id) {
    try {
      const apiKey = await getUserProviderKey(
        db, userId, task.project_id as string, task.provider as string,
      )
      if (apiKey) {
        const processor = ProcessorRegistry.get(task.type as AsyncTaskType)
        await processor.cancel(task.external_task_id as string, apiKey)
      }
    } catch {
      // Provider 取消失败不阻塞用户操作
      console.warn(`Provider cancel failed for task ${params.id}`)
    }
  }

  // ----------------------------------------------------------
  // 更新状态 + 退还积分
  // ----------------------------------------------------------
  const now = Date.now()
  await db
    .prepare(
      `UPDATE async_tasks
       SET status = 'cancelled', error_message = '用户主动取消', completed_at = ?
       WHERE id = ?`,
    )
    .bind(now, params.id)
    .run()

  const credits = task.credits_charged as number
  await refundCredits(db, userId, credits, params.id)

  return Response.json({
    success: true,
    message: '任务已取消',
    creditsRefunded: credits,
  })
}
```

### 3.6 并发控制

简单直接：Free 用户最多 1 个并发任务，Pro 用户最多 4 个。无优先级队列。

```typescript
// ============================================================
//  lib/concurrency.ts — 并发控制
//  Method C 的并发模型: 简单计数，无优先级
// ============================================================

const PLAN_LIMITS: Record<string, number> = {
  free: 1,
  pro: 4,
}

/**
 * 检查用户是否可以提交新任务
 *
 * 并发 = 当前 pending + running 的任务数
 * 注意: cancelled/completed/failed 不计入并发
 */
async function checkConcurrency(
  db: D1Database,
  userId: string,
): Promise<{
  allowed: boolean
  reason?: string
  currentCount: number
  maxCount: number
}> {
  // 获取用户套餐
  const sub = await db
    .prepare('SELECT plan FROM subscriptions WHERE user_id = ? AND status = ?')
    .bind(userId, 'active')
    .first<{ plan: string }>()

  const plan = sub?.plan ?? 'free'
  const maxCount = PLAN_LIMITS[plan] ?? 1

  // 统计活跃任务
  const result = await db
    .prepare(
      `SELECT COUNT(*) as count FROM async_tasks
       WHERE user_id = ? AND status IN ('pending', 'running')`,
    )
    .bind(userId)
    .first<{ count: number }>()

  const currentCount = result?.count ?? 0

  if (currentCount >= maxCount) {
    return {
      allowed: false,
      reason: `并发任务数已达上限 (${currentCount}/${maxCount})，请等待当前任务完成或升级套餐`,
      currentCount,
      maxCount,
    }
  }

  return { allowed: true, currentCount, maxCount }
}
```

### 3.7 重试机制

重试在 `checkTask`（3.4 节）中自然发生：当 Provider 返回 failed 且 retry_count < max_retries 时，重新调用 `processor.submit`。

```typescript
// ============================================================
//  重试策略 — 嵌入 checkTask 流程
// ============================================================

/**
 * 重试决策树 (在 checkTask 内部执行):
 *
 * Provider 返回 failed
 *   ├── retryable === false → 最终失败 (不可重试的错误)
 *   ├── retry_count >= max_retries → 最终失败 (重试耗尽)
 *   └── otherwise → 重新 processor.submit()
 *       ├── submit 成功 → status=running, retry_count++, 新 externalTaskId
 *       └── submit 失败 → 保持 running, retry_count++, 等下次轮询
 *
 * 不可重试错误 (retryable: false):
 *   - 内容违规 (content_policy_violation)
 *   - 无效参数 (invalid_params)
 *   - 认证失败 (unauthorized) — 用户 API Key 无效
 *   - 配额耗尽 (quota_exceeded) — 用户自己的 Provider 配额
 */

interface CheckResult {
  status: 'running' | 'completed' | 'failed'
  progress?: number          // 0-100
  output?: TaskOutput        // completed 时必须提供
  error?: string             // failed 时的错误信息
  retryable?: boolean        // 默认 true，明确不可重试时设为 false
}
```

### 4.1 处理器接口设计

```typescript
// ============================================================
//  处理器接口 — Method C 版本
//  lib/processors/types.ts
//
//  与 v1 (Queues) 的关键区别:
//  - 移除 execute() 单一入口（不再有 Consumer Worker 驱动完整生命周期）
//  - 拆分为 submit / checkStatus / cancel 三个独立方法
//  - 接受用户的 apiKey 参数（用户自带 Key）
//  - 各 Processor 内部负责 R2 上传（不再由基类统一处理）
// ============================================================

interface SubmitInput {
  provider: string
  modelId: string
  input: Record<string, unknown>
  taskId: string
  userId: string
}

interface SubmitResult {
  externalTaskId: string     // Provider 返回的任务 ID
}

interface TaskOutput {
  files: Array<{
    url: string              // R2 公开 URL
    type: 'video' | 'image' | 'audio'
    filename: string
    size?: number
    thumbnailUrl?: string
  }>
  metadata?: Record<string, unknown>
}

interface CheckResult {
  status: 'running' | 'completed' | 'failed'
  progress?: number          // 0-100
  output?: TaskOutput        // completed 时提供
  error?: string             // failed 时提供
  retryable?: boolean        // 默认 true
}

/**
 * 任务处理器接口
 *
 * 每个 AI Provider 对应一个 Processor 实现。
 * Processor 是无状态的——所有状态存储在 D1 中。
 */
interface TaskProcessor {
  /** 处理器支持的任务类型 */
  readonly taskType: AsyncTaskType

  /** 处理器支持的 Provider 名称 */
  readonly provider: string

  /**
   * 提交任务到外部 Provider
   * @returns externalTaskId — Provider 返回的任务标识
   * @throws 提交失败时抛出异常
   */
  submit(input: SubmitInput, apiKey: string): Promise<SubmitResult>

  /**
   * 查询外部任务状态
   * 懒评估: 仅在前端轮询时被调用
   * 如果 completed，Processor 内部负责下载结果并上传到 R2
   */
  checkStatus(externalTaskId: string, apiKey: string): Promise<CheckResult>

  /**
   * 尝试取消外部任务（best effort）
   * 部分 Provider 可能不支持取消，此时静默忽略
   */
  cancel(externalTaskId: string, apiKey: string): Promise<void>
}
```

### 4.5 处理器注册表

```typescript
// ============================================================
//  处理器注册表
//  lib/processors/registry.ts
//
//  按 (taskType, provider) 二元组注册 Processor
//  支持同一 taskType 下多个 Provider
// ============================================================

class ProcessorRegistry {
  private static processors = new Map<string, TaskProcessor>()

  static {
    // 视频生成
    this.register(new KlingVideoProcessor())
    this.register(new WanVideoProcessor())

    // 图片生成
    this.register(new QwenImageProcessor())
    // this.register(new ZImageProcessor())

    // 音频生成
    this.register(new AudioGenProcessor())
  }

  private static register(processor: TaskProcessor): void {
    const key = `${processor.taskType}:${processor.provider}`
    this.processors.set(key, processor)
  }

  /**
   * 获取处理器
   *
   * 查找顺序: 精确匹配 taskType:provider → 回退到 taskType 的任意 provider
   */
  static get(taskType: AsyncTaskType, provider?: string): TaskProcessor {
    if (provider) {
      const exact = this.processors.get(`${taskType}:${provider}`)
      if (exact) return exact
    }

    // 回退: 找该 taskType 的第一个 processor
    for (const [key, processor] of this.processors) {
      if (key.startsWith(`${taskType}:`)) return processor
    }

    throw new Error(`未注册的任务处理器: ${taskType}${provider ? `:${provider}` : ''}`)
  }
}
```

### 5.1 useTaskPolling Hook

```typescript
// ============================================================
//  hooks/api/useTaskPolling.ts
//  TanStack Query 驱动的任务状态轮询
//
//  核心设计:
//  - 活跃状态按 TASK_CONFIG[taskType].pollIntervalMs 轮询
//  - 终态自动停止轮询
//  - 后台标签页不轮询 (refetchIntervalInBackground: false)
//  - 窗口聚焦时立即刷新
// ============================================================

import { useQuery } from '@tanstack/react-query'

interface TaskDetail {
  id: string
  type: AsyncTaskType
  status: TaskStatus
  progress: number
  provider: string
  modelId: string
  inputData: Record<string, unknown>
  outputData: TaskOutput | null
  creditsCharged: number
  errorMessage: string | null
  retryCount: number
  createdAt: number
  startedAt: number | null
  completedAt: number | null
}

/**
 * 轮询单个任务状态
 *
 * Method C 的关键: 每次 GET 请求都可能触发服务端查询 Provider
 * 因此轮询间隔本身就是 Provider 查询频率的上限
 */
export function useTaskPolling(taskId: string | null, taskType: AsyncTaskType) {
  return useQuery({
    queryKey: ['tasks', 'detail', taskId],
    queryFn: async (): Promise<TaskDetail> => {
      const response = await fetch(`/api/tasks/${taskId}`)
      if (!response.ok) throw new Error('获取任务状态失败')
      return response.json()
    },
    enabled: !!taskId,
    refetchInterval: (query) => {
      const data = query.state.data
      if (!data) return TASK_CONFIG[taskType].pollIntervalMs

      // 终态: 停止轮询
      if (isTerminal(data.status)) return false

      // 活跃态: 按任务类型配置的间隔轮询
      return TASK_CONFIG[taskType].pollIntervalMs
    },
    refetchIntervalInBackground: false,  // 后台标签页不轮询
    refetchOnWindowFocus: true,           // 窗口聚焦时立即刷新
    staleTime: 2000,
    gcTime: 5 * 60_000,
  })
}

/**
 * 轮询用户所有活跃任务

 * 用于全局任务面板 / 通知栏
 */
export function useActiveTasks() {

      const response = await fetch('/api/tasks?status=active')
      if (!response.ok) throw new Error('获取任务列表失败')
```

### 5.2 任务进度组件

```typescript

  taskType,
  onComplete,
  onFailed,

      onComplete?.(task.outputData)
    }
    if (task?.status === 'failed' && task.errorMessage) {
      onFailed?.(task.errorMessage)
    }
  }, [task?.status])

  if (isLoading || !task) return <Spinner />

  return (
    <div className="task-progress">
      <div className="flex items-center justify-between">
        <span className="text-sm text-gray-600">
          {STATUS_LABELS[task.status]}
        </span>
        <span className="text-sm font-mono">
          {task.progress}%
        </span>
      </div>
      <div className="w-full bg-gray-200 rounded h-2 mt-1">
        <div
          className="bg-indigo-500 rounded h-2 transition-all duration-500"
          style={{ width: `${task.progress}%` }}
        />
      </div>
      {task.errorMessage && (
        <p className="text-sm text-red-500 mt-1">{task.errorMessage}</p>
      )}
    </div>
  )
}

const STATUS_LABELS: Record<string, string> = {
  pending:   '准备中...',
  running:   '生成中...',
  completed: '已完成',
  failed:    '生成失败',
  cancelled: '已取消',
}
```

## 六、积分集成

### 6.1 三阶段积分模型

Method C 的积分流程：**冻结 → 确认/退还**。

```
submit (POST /api/tasks)
    │
    ├── freezeCredits(amount)        ← 从余额中冻结积分
    │
    ▼
checkTask (GET /api/tasks/:id) × N 次轮询
    │
    ├── completed → confirmCredits() ← 最终扣减，冻结 → 已消费
    ├── failed(final) → refundCredits() ← 退还冻结积分
    └── cancelled → refundCredits()  ← 退还冻结积分
```

### 6.2 积分操作实现

```typescript
// ============================================================
//  lib/credits.ts — 积分三阶段模型
// ============================================================

/**
 * 冻结积分（submit 时调用）
 *
 * 从 balance 中扣除，加入 frozen_credits
 * 如果余额不足，返回 success: false
 */
async function freezeCredits(
  db: D1Database,
  userId: string,
  amount: number,
): Promise<{ success: boolean; balance: number }> {
  const balance = await db
    .prepare('SELECT balance FROM credit_balances WHERE user_id = ?')
    .bind(userId)
    .first<{ balance: number }>()

  const currentBalance = balance?.balance ?? 0
  if (currentBalance < amount) {
    return { success: false, balance: currentBalance }
  }

  await db
    .prepare(
      `UPDATE credit_balances
       SET balance = balance - ?, frozen_credits = frozen_credits + ?, updated_at = ?
       WHERE user_id = ?`,
    )
    .bind(amount, amount, Date.now(), userId)
    .run()

  // 记录冻结交易
  await db
    .prepare(
      `INSERT INTO credit_transactions (id, user_id, type, amount, balance_after, source, reference_id, description, created_at)
       VALUES (?, ?, 'freeze', ?, ?, 'ai_task', ?, ?, ?)`,
    )
    .bind(
      nanoid(), userId, -amount, currentBalance - amount,
      null, '任务积分冻结', Date.now(),
    )
    .run()

  return { success: true, balance: currentBalance - amount }
}

/**
 * 确认消费（checkTask 发现 completed 时调用）
 *
 * 从 frozen_credits 中扣除，积分最终消费
 */
async function confirmCredits(
  db: D1Database,
  userId: string,
  amount: number,
  taskId: string,
): Promise<void> {
  if (amount <= 0) return

  await db
    .prepare(
      `UPDATE credit_balances
       SET frozen_credits = frozen_credits - ?, total_spent = total_spent + ?, updated_at = ?
       WHERE user_id = ?`,
    )
    .bind(amount, amount, Date.now(), userId)
    .run()

  await db
    .prepare(
      `INSERT INTO credit_transactions (id, user_id, type, amount, balance_after, source, reference_id, description, created_at)
       VALUES (?, ?, 'consume', ?, (SELECT balance FROM credit_balances WHERE user_id = ?), 'ai_task', ?, ?, ?)`,
    )
    .bind(
      nanoid(), userId, -amount, userId,
      taskId, `任务完成积分消费 (${taskId})`, Date.now(),
    )
    .run()
}

/**
 * 退还积分（checkTask 发现 failed/cancelled 时调用）
 *
 * 从 frozen_credits 中移回 balance
 */
async function refundCredits(
  db: D1Database,
  userId: string,
  amount: number,
  taskId: string,
): Promise<void> {
  if (amount <= 0) return

  await db
    .prepare(
      `UPDATE credit_balances
       SET balance = balance + ?, frozen_credits = frozen_credits - ?, updated_at = ?
       WHERE user_id = ?`,
    )
    .bind(amount, amount, Date.now(), userId)
    .run()

  await db
    .prepare(
      `INSERT INTO credit_transactions (id, user_id, type, amount, balance_after, source, reference_id, description, created_at)
       VALUES (?, ?, 'refund', ?, (SELECT balance FROM credit_balances WHERE user_id = ?), 'refund', ?, ?, ?)`,
    )
    .bind(
      nanoid(), userId, amount, userId,
      taskId, `任务失败退还积分 (${taskId})`, Date.now(),
    )
    .run()
}
```

```typescript
async function downloadToR2(
  sourceUrl: string,
  r2Key: string,
): Promise<string> {
  // 从 Provider 下载文件
  const response = await fetch(sourceUrl)
  if (!response.ok) throw new Error(`下载失败: ${response.status}`)

  const contentType = response.headers.get('content-type') ?? 'application/octet-stream'
  const body = await response.arrayBuffer()

  // 上传到 R2
  await env.BUCKET.put(r2Key, body, {
    httpMetadata: { contentType },
  })

  return `${env.R2_PUBLIC_URL}/${r2Key}`
}
```

```typescript
// ============================================================
//  GET /api/tasks — 查询用户任务列表
//  app/api/tasks/route.ts (GET handler)
// ============================================================

export async function GET(request: Request) {
  const { userId } = auth()
  if (!userId) return Response.json({ error: 'Unauthorized' }, { status: 401 })

  const url = new URL(request.url)
  const status = url.searchParams.get('status') ?? 'all'
  const type = url.searchParams.get('type')
  const page = Math.max(1, parseInt(url.searchParams.get('page') ?? '1'))
  const pageSize = Math.min(
    50,
    Math.max(1, parseInt(url.searchParams.get('pageSize') ?? '20')),
  )
  const offset = (page - 1) * pageSize

  // 构建查询条件
  let whereClause = 'WHERE user_id = ?'
  const bindings: unknown[] = [userId]

  if (status === 'active') {
    whereClause += " AND status IN ('pending', 'running')"
  } else if (status === 'completed') {
    whereClause += " AND status = 'completed'"
  } else if (status === 'failed') {
    whereClause += " AND status IN ('failed', 'cancelled')"
  }

  if (type) {
    whereClause += ' AND type = ?'
    bindings.push(type)
  }

  // 查询总数
  const countResult = await db
    .prepare(`SELECT COUNT(*) as total FROM async_tasks ${whereClause}`)
    .bind(...bindings)
    .first<{ total: number }>()
  const total = countResult?.total ?? 0

  // 查询数据
  const tasks = await db
    .prepare(
      `SELECT * FROM async_tasks ${whereClause}
       ORDER BY created_at DESC
       LIMIT ? OFFSET ?`,
    )
    .bind(...bindings, pageSize, offset)
    .all()

  return Response.json({
    tasks: tasks.results.map(formatTaskResponse),
    total,
    page,
    pageSize,
    hasMore: offset + pageSize < total,
  } satisfies TaskListResponse)
}
```

### 10.1 核心指标

| 指标         | 计算方式                           | 健康阈值 | 告警阈值 |
| ------------ | ---------------------------------- | -------- | -------- |
| 任务成功率   | completed / (completed + failed)   | > 95%    | < 90%    |
| 平均处理时间 | avg(completed_at - created_at)     | < 5min   | > 15min  |
| 活跃任务数   | count(status IN pending, running)  | < 50     | > 200    |
| 重试率       | sum(retry_count > 0) / total       | < 10%    | > 25%    |

### 10.2 监控查询

```sql
--  监控仪表盘 SQL 查询 — Method C 适配
-- ============================================================

-- 过去 24 小时任务成功率
SELECT
  ROUND(
    COUNT(CASE WHEN status = 'completed' THEN 1 END) * 100.0 /
    NULLIF(COUNT(CASE WHEN status IN ('completed', 'failed') THEN 1 END), 0),
    2
  ) AS success_rate
FROM async_tasks
WHERE created_at > ? - 86400000;

-- 过去 24 小时平均处理时间（毫秒）
SELECT
  type,
  AVG(completed_at - created_at) AS avg_duration_ms,
  MIN(completed_at - created_at) AS min_duration_ms,
  MAX(completed_at - created_at) AS max_duration_ms,
  COUNT(*) AS count
FROM async_tasks
WHERE status = 'completed'
  AND created_at > ? - 86400000
GROUP BY type;

-- 当前活跃任务
SELECT
  status,
  COUNT(*) AS count
FROM async_tasks
WHERE status IN ('pending', 'running')
GROUP BY status;

-- 过去 1 小时按提供商分布的失败率
SELECT
  provider,
  COUNT(*) AS total,
  COUNT(CASE WHEN status = 'failed' THEN 1 END) AS failures,
  ROUND(
    COUNT(CASE WHEN status = 'failed' THEN 1 END) * 100.0 / COUNT(*),
    2
  ) AS failure_rate
FROM async_tasks
WHERE created_at > ? - 3600000
GROUP BY provider;
```

## 附录：Method C 架构决策记录

### A.1 为什么不用 Cloudflare Queues

**直接原因**：Cloudflare Queues 是付费服务（$0.40/百万消息），对于 MVP 阶段和中小规模场景是不必要的成本。

**架构原因**：

- 引入 Queues 需要独立的 Consumer Worker，增加部署单元和运维复杂度
- Producer → Queue → Consumer 的异步消息链路增加了调试难度
- 消息格式 (TaskMessage) 需要与 D1 记录保持同步，增加了数据一致性的维护成本
- 死信队列、批量消费等概念引入了不必要的复杂性

**本质判断**：对于日均 <1000 次任务的场景，Queues 是过度工程。D1 的 SELECT/UPDATE 完全能承载这个量级。

### A.2 为什么不用 Cron Trigger

Cron Trigger 方案：定时扫描 D1 中 running 状态的任务，批量查询 Provider 状态。

**问题**：

- **空转浪费**：没有任务时 Cron 仍会触发，浪费 Worker 调用次数
- **实时性受限**：Cron 最短间隔 1 分钟，用户体验不如客户端驱动的秒级轮询
- **批量查询复杂**：需要处理不同 Provider、不同用户的 API Key，批量逻辑复杂

**Method C 的优势**：懒评估——只有用户在看的时候才查询 Provider，完美匹配"按需"的哲学。

### A.3 为什么不用客户端直连 Provider

让前端直接调用 Kling/Wan/QWEN 的 API：

**不可行**：

- **CORS 限制**：大部分 AI Provider API 不允许浏览器端直接调用
- **API Key 暴露**：用户的 API Key 会出现在前端代码中，存在安全风险
- **无法统一管理**：积分扣减、并发控制、重试逻辑无法在前端可靠实现

**Method C 的做法**：服务端代理——前端只与我们的 API Route 通信，API Route 代理调用 Provider。

### A.4 唯一权衡：Provider 结果过期风险

**风险场景**：用户提交任务后关闭浏览器，长时间不回来。Provider 完成了任务但结果 URL 有过期时间（通常 24h）。由于无人轮询，API Route 不会被触发去下载结果。

**当前策略 (MVP)**：接受这个风险。理由：
- 用户提交 AI 生成任务后通常会关注结果
- 即使过期，积分在超时检测后会被退还
- 损失的只是 Provider 端的计算资源（用户自己的 API 配额）

**后期兜底**：当用户量增长后，可以添加一个轻量级 Cron Trigger（每 10 分钟），扫描 running 且接近超时的任务，主动查询 Provider 并下载结果。这是一个低成本的增量改进，不影响 Method C 的核心架构。
