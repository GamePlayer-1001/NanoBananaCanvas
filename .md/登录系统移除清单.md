# 登录系统移除清单

> 目标：移除当前 Clerk 登录系统，把项目收敛为“先能跑核心画板，再重做身份系统”的状态。

## 一、拆除目标

1. 移除现有 Clerk 依赖与身份感知 UI。
2. 解除路由保护，让核心页面可以在无登录前提下访问。
3. 去掉 API 对 Clerk 用户镜像的强耦合。
4. 为未来重做登录保留清晰边界，而不是继续沿用当前污染过的集成方式。

## 二、拆除范围

### 2.1 路由与中间件

- `apps/web/middleware.ts`
- `apps/web/components/clerk-provider.tsx`
- 所有路由级 `AppClerkProvider` 包装
- `apps/web/app/[locale]/(auth)/*`

### 2.2 身份 UI

- `apps/web/components/auth/*`
- `apps/web/components/canvas/canvas-top-toolbar.tsx` 中的 `UserButton`
- `apps/web/components/layout/app-sidebar.tsx` 中的 `useUser`、头像与用户态显示
- `apps/web/components/profile/*` 中所有依赖 Clerk 的面板

### 2.3 API 认证与用户同步

- `apps/web/lib/api/auth.ts`
- `apps/web/app/api/users/me/route.ts`
- `apps/web/app/api/webhooks/clerk/route.ts`
- 所有依赖 `requireAuth` / `optionalAuth` 的路由，需要改成匿名模式或本地模式

### 2.4 数据模型

- `users` 表中的 `clerk_id` 强依赖
- 所有以 `user_id` 作为硬前提的工作流/文件夹/任务读写逻辑

## 三、执行策略

1. 先把“必须登录才能访问”的中间件保护移除。
2. 把 UI 中所有 Clerk 组件替换为匿名态占位或直接删除。
3. 把 API 从“强制登录”改为“本地单用户模式”或“临时匿名工作区模式”。
4. 最后再删除 Clerk 包与 webhook。

## 四、推荐过渡方案

### 方案 A：本地单用户模式

- 工作流、文件夹、设置全部视作单用户资源
- 不再区分 `user_id`
- 最适合当前“先把产品做通”的阶段

### 方案 B：匿名会话模式

- 使用浏览器本地会话 ID
- 仅做轻量隔离，不做真实账户系统
- 适合未来要快速接回登录，但当前仍需区分会话的阶段

当前更推荐：**方案 A**

原因：最简单、边界最干净、对当前开发目标最诚实。

## 五、关键风险

1. `workflows`、`folders`、`async_tasks` 都默认依赖 `user_id`，移除登录不等于直接删 Clerk 包，必须先定义新的资源归属策略。
2. 自动保存、上传、任务提交、历史记录都可能在服务端查询 `user_id`，需要统一改造，否则会出现“页面能开，功能全 401”的假通畅。
3. 公开页与应用页当前由中间件共同处理 locale 与 auth，拆登录时要防止顺手把 i18n 路由也打坏。

## 六、验收清单

- [ ] 项目中不再存在 `@clerk/nextjs` 与 `@clerk/localizations` 的运行时引用
- [ ] 中间件不再包含路由保护与 Clerk 代理逻辑
- [ ] `/sign-in`、`/sign-up` 路由已移除或重定向到新的说明页
- [ ] 画板、工作区、核心 API 在无登录前提下可访问
- [ ] 不再有 `Clerk user not found`、`AUTH_UNAUTHORIZED` 作为核心功能阻断
- [ ] `pnpm lint`、`pnpm test` 不再引用 Clerk 相关模块

## 七、完成定义

完成不等于“看不到登录页”，而等于：

1. 核心产品不再依赖第三方身份系统才能使用。
2. 资源归属逻辑已经被新的单用户/匿名模式替代。
3. 重新接回登录时，不需要先清理旧 Clerk 遗留耦合。
