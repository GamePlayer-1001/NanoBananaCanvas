## 一、项目概述

### 1.1 项目名称

Nano Banana Canvas

### 1.2 项目定位

可视化 AI 工作流编排平台，让用户通过拖拽节点、连接管道的方式，组合各类 AI 能力，构建自动化工作流。同时提供社区功能，用户可分享、发现、克隆他人的工作流。

### 1.3 目标用户

- AI 爱好者、创作者
- 需要批量处理 AI 任务的用户
- 希望低代码方式使用 AI 的非技术用户
- 希望学习和复用他人工作流的用户

### 1.4 参考产品

- n8n（工作流自动化）
- ComfyUI（AI 图像工作流）
- Dify（AI 应用构建平台）
- RunningHub（AI 工作流社区）

## 四、关键技术点分析

### 4.1 Clerk 集成

**问题**：需要集成第三方用户认证系统

**解决方案**：

```typescript
// 前端：ClerkProvider 包裹应用
import { ClerkProvider, SignIn, SignUp, useUser } from '@clerk/clerk-react'

function App() {
  return (
    <ClerkProvider publishableKey={CLERK_PUBLISHABLE_KEY}>
      <RouterProvider router={router} />
    </ClerkProvider>
  )
}

// 后端：验证 Clerk JWT
import { verifyToken } from '@clerk/backend'

async function authMiddleware(c, next) {
  const token = c.req.header('Authorization')?.replace('Bearer ', '')
  if (!token) return c.json({ error: 'Unauthorized' }, 401)

  try {
    const payload = await verifyToken(token, {
      secretKey: c.env.CLERK_SECRET_KEY
    })
    c.set('userId', payload.sub)
    return next()
  } catch {
    return c.json({ error: 'Invalid token' }, 401)
  }
}
```

**可行性**：✅ Clerk 官方支持 React 和 Cloudflare Workers

### 4.2 Clerk 用户信息同步策略

**问题**：Clerk 管理用户认证，但我们的数据库需要存储用户信息以关联工作流等业务数据

**解决方案**：采用**双重同步策略**，方案一和方案二并行使用

#### 方案一：首次登录时自动创建用户记录（优先实现）

```
用户通过 Clerk 登录
       ↓
前端调用 GET /api/users/me
       ↓
后端检查数据库是否有该用户
       ↓
  ┌────┴────┐
  ↓         ↓
没有       有
  ↓         ↓
创建记录   直接返回
```

```typescript
// 后端 API：GET /api/users/me
app.get('/api/users/me', authMiddleware, async (c) => {
  const clerkUserId = c.get('userId')
  const clerkUser = c.get('user') // Clerk 提供的用户信息

  // 查询数据库
  let user = await c.env.DB.prepare(
    'SELECT * FROM users WHERE id = ?'
  ).bind(clerkUserId).first()

  // 不存在则创建
  if (!user) {
    await c.env.DB.prepare(`
      INSERT INTO users (id, email, name, avatar_url, created_at, updated_at)
      VALUES (?, ?, ?, ?, ?, ?)
    `).bind(
      clerkUserId,
      clerkUser.email,
      clerkUser.name || clerkUser.username,
      clerkUser.imageUrl,
      Date.now(),
      Date.now()
    ).run()

    user = await c.env.DB.prepare(
      'SELECT * FROM users WHERE id = ?'
    ).bind(clerkUserId).first()
  }

  return c.json({ success: true, data: user })
})
```

**优点**：
- 实现简单，不需要额外配置
- 不依赖外部服务
- 适合快速上线

**缺点**：
- 用户信息可能不及时（用户在 Clerk 改了头像，但没访问网站，数据库里还是旧的）

#### 方案二：通过 Webhook 实时同步（后续增强）

```
Clerk 用户事件发生（注册/更新/删除）
       ↓
Clerk 推送 Webhook 到后端
       ↓
POST /api/webhooks/clerk
       ↓
后端根据事件类型处理
       ↓
  ┌────┴────┬────────┐
  ↓         ↓        ↓
user.created  user.updated  user.deleted
  ↓         ↓        ↓
创建记录   更新记录   删除记录
```

```typescript
// 后端 API：POST /api/webhooks/clerk
import { Webhook } from 'svix'

app.post('/api/webhooks/clerk', async (c) => {
  const payload = await c.req.text()
  const headers = c.req.header()

  // 验证 Webhook 签名
  const wh = new Webhook(c.env.CLERK_WEBHOOK_SECRET)
  const evt = wh.verify(payload, {
    'svix-id': headers['svix-id'],
    'svix-timestamp': headers['svix-timestamp'],
    'svix-signature': headers['svix-signature'],
  })

  // 根据事件类型处理
  switch (evt.type) {
    case 'user.created':
      await c.env.DB.prepare(`
        INSERT INTO users (id, email, name, avatar_url, created_at, updated_at)
        VALUES (?, ?, ?, ?, ?, ?)
      `).bind(
        evt.data.id,
        evt.data.email_addresses[0]?.email_address,
        evt.data.first_name || evt.data.username,
        evt.data.image_url,
        Date.now(),
        Date.now()
      ).run()
      break

    case 'user.updated':
      await c.env.DB.prepare(`
        UPDATE users SET email = ?, name = ?, avatar_url = ?, updated_at = ?
        WHERE id = ?
      `).bind(
        evt.data.email_addresses[0]?.email_address,
        evt.data.first_name || evt.data.username,
        evt.data.image_url,
        Date.now(),
        evt.data.id
      ).run()
      break

    case 'user.deleted':
      await c.env.DB.prepare('DELETE FROM users WHERE id = ?')
        .bind(evt.data.id).run()
      break
  }

  return c.json({ received: true })
})
```

**优点**：
- 用户信息实时同步，始终与 Clerk 保持一致
- 可以捕获用户删除账号等事件

**缺点**：
- 需要在 Clerk Dashboard 配置 Webhook URL
- 本地开发时需要用 ngrok 等工具暴露端口
- 需要额外的签名验证逻辑

#### 两者结合的最佳实践

| 场景 | 方案一处理 | 方案二处理 |
|-----|----------|----------|
| 用户首次登录 | ✅ 创建记录 | ✅ 也会触发（双保险） |
| 用户修改头像/昵称 | ❌ 不知道 | ✅ 实时更新 |
| 用户删除账号 | ❌ 不知道 | ✅ 删除记录 |
| 用户长期不访问但改了资料 | ❌ 下次访问才同步 | ✅ 立即同步 |

**推荐实施顺序**：
1. **P1 阶段**：先实现方案一 → 快速上线，满足基本需求
2. **后续优化**：加上方案二 → 解决数据一致性问题

**可行性**：✅ 两种方案都有成熟实现，可并行使用

### 4.3 双轨路由架构（Landing Page）

**问题**：未登录用户和已登录用户需要看到完全不同的页面体系

**解决方案**：使用 React Router + Clerk 实现根路由守卫分流

```typescript
// RootGuard.tsx - 根路由守卫
function RootGuard() {
  const { isSignedIn, isLoaded } = useAuth()

  if (!isLoaded) return <LoadingSpinner />

  // 已登录 → 重定向到 /workspace
  // 未登录 → 显示 Landing Page
  return isSignedIn
    ? <Navigate to="/workspace" replace />
    : <Outlet />
}

// router/index.tsx
export const router = createBrowserRouter([
  {
    path: '/',
    element: <RootGuard />,
    children: [
      {
```

```typescript
function GlobalSearch() {
  const navigate = useNavigate()
  const [query, setQuery] = useState('')

  const handleSearch = () => {
    if (query.trim()) {
      navigate(`/explore?q=${encodeURIComponent(query)}`)
    }
  }

  return (
    <div className="flex items-center">
      <input
        value={query}
        onChange={(e) => setQuery(e.target.value)}
        onKeyDown={(e) => e.key === 'Enter' && handleSearch()}
        placeholder="搜索作品广场..."
      />
      <button onClick={handleSearch}>搜索</button>
    </div>
  )
}
```

### 4.5 首次访问登录弹窗

**问题**：首次访问时弹窗提示登录，但用户可以关闭

**解决方案**：

```typescript
function FirstVisitModal() {
  const [show, setShow] = useState(false)
  const { isSignedIn } = useUser()

  useEffect(() => {
    const hasVisited = localStorage.getItem('hasVisited')
    if (!hasVisited && !isSignedIn) {
      setShow(true)
      localStorage.setItem('hasVisited', 'true')
    }
  }, [isSignedIn])

  if (!show) return null

  return (
    <Modal onClose={() => setShow(false)}>
      <h2>欢迎来到 Nano Banana Canvas</h2>
      <p>登录后可以创建和分享您的工作流</p>
      <SignIn />
      <button onClick={() => setShow(false)}>稍后再说</button>
    </Modal>
  )
}
```

**可行性**：✅ localStorage + Clerk 组件即可实现

### 4.6 工作流克隆

**问题**：用户克隆他人工作流到自己的创作空间

**解决方案**：

```typescript
// 后端 API
app.post('/api/workflows/:id/clone', authMiddleware, async (c) => {
  const { id } = c.req.param()
  const userId = c.get('userId')

  // 获取原工作流
  const original = await c.env.DB.prepare(
    'SELECT * FROM workflows WHERE id = ? AND is_public = 1'
  ).bind(id).first()

  if (!original) return c.json({ error: 'Not found' }, 404)

  // 创建副本
  const newId = generateId()
  await c.env.DB.prepare(`
    INSERT INTO workflows (id, user_id, name, data, created_at, updated_at)
    VALUES (?, ?, ?, ?, ?, ?)
  `).bind(
    newId,
    userId,
    `${original.name} (副本)`,
    original.data,
    Date.now(),
    Date.now()
  ).run()

  // 更新克隆计数
  await c.env.DB.prepare(
    'UPDATE workflows SET clone_count = clone_count + 1 WHERE id = ?'
  ).bind(id).run()

  return c.json({ id: newId })
})
```

**可行性**：✅ 标准数据库操作，无技术障碍
