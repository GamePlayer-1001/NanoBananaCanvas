## 一、设计目标

P3 阶段将实现实时协作功能。本文档定义 **P1 阶段必须预留的数据模型和接口设计**，确保 P3 实现时不需要破坏性迁移。

**原则**：P1 不实现协作功能，但数据模型从第一天就兼容多用户。

## 二、数据模型预留

### 2.1 工作流表 — 添加 workspace 概念

```sql
-- ============================================
-- P1 创建时的 workflows 表
-- 已包含 P3 协作所需字段（nullable）
-- ============================================
CREATE TABLE workflows (
  id TEXT PRIMARY KEY,
  user_id TEXT NOT NULL,             -- 创建者（永远不变）
  workspace_id TEXT,                 -- P3: 所属工作空间（P1 为 NULL = 个人空间）
  name TEXT NOT NULL,
  description TEXT,
  category_id TEXT,
  data TEXT NOT NULL,                -- JSON: 工作流数据
  thumbnail_url TEXT,
  is_public INTEGER DEFAULT 0,
  like_count INTEGER DEFAULT 0,
  favorite_count INTEGER DEFAULT 0,
  clone_count INTEGER DEFAULT 0,
  version INTEGER DEFAULT 1,         -- P3: 乐观并发控制版本号
  last_edited_by TEXT,               -- P3: 最后编辑者 user_id
  created_at INTEGER,
  updated_at INTEGER,
  published_at INTEGER,
  FOREIGN KEY (user_id) REFERENCES users(id)
);

-- P1 索引
CREATE INDEX idx_workflows_user ON workflows(user_id);
CREATE INDEX idx_workflows_public ON workflows(is_public, published_at);
-- P3 索引（现在创建不影响性能，但 P3 需要时已就绪）
CREATE INDEX idx_workflows_workspace ON workflows(workspace_id);
```

### 2.2 工作空间表（P3 创建，P1 预定义结构）

```sql
-- ============================================
-- P3 实现时创建
-- P1 阶段仅作为文档记录，不实际建表
-- ============================================

-- 工作空间
CREATE TABLE workspaces (
  id TEXT PRIMARY KEY,
  name TEXT NOT NULL,
  owner_id TEXT NOT NULL,            -- 工作空间拥有者
  plan TEXT DEFAULT 'team',          -- 'team' | 'enterprise'
  max_members INTEGER DEFAULT 5,
  created_at INTEGER,
  updated_at INTEGER,
  FOREIGN KEY (owner_id) REFERENCES users(id)
);

-- 工作空间成员
CREATE TABLE workspace_members (
  id TEXT PRIMARY KEY,
  workspace_id TEXT NOT NULL,
  user_id TEXT NOT NULL,
  role TEXT NOT NULL DEFAULT 'editor',  -- 'owner' | 'admin' | 'editor' | 'viewer'
  invited_by TEXT,
  joined_at INTEGER,
  UNIQUE(workspace_id, user_id),
  FOREIGN KEY (workspace_id) REFERENCES workspaces(id),
  FOREIGN KEY (user_id) REFERENCES users(id)
);

-- 协作编辑锁（防冲突）
CREATE TABLE editing_locks (
  workflow_id TEXT PRIMARY KEY,
  user_id TEXT NOT NULL,
  locked_at INTEGER NOT NULL,
  expires_at INTEGER NOT NULL,       -- 锁自动过期（防止僵尸锁）
  FOREIGN KEY (workflow_id) REFERENCES workflows(id)
);
```

## 四、实时协作技术选型预研（P3 参考）

### 4.1 方案对比

| 方案                           | 优势                                   | 劣势                   | 适用性                        |
| ------------------------------ | -------------------------------------- | ---------------------- | ----------------------------- |
| **Cloudflare Durable Objects** | 边缘运行、强一致性、WebSocket 原生支持 | API 学习曲线、调试困难 | **推荐** — 与现有基础设施一致 |
| Liveblocks                     | 开箱即用、React 集成好                 | 第三方依赖、价格       | 备选                          |
| Yjs + WebSocket                | 开源、CRDT 保证冲突解决                | 需要自建 WS 服务器     | 备选                          |
| Supabase Realtime              | Postgres 变更推送                      | 不用 Supabase 就不适用 | 不适用                        |

### 4.2 推荐方案：Durable Objects

```
用户 A 编辑 ──WebSocket──┐
                          │
用户 B 编辑 ──WebSocket──├──► Durable Object (每个 workflow 一个实例)
                          │        │
用户 C 查看 ──WebSocket──┘        │
                                   ▼
                              D1 持久化（定期快照）
```

**P1 准备**：

- 工作流 data 字段使用 JSON，结构设计为可被 CRDT 操作的粒度
- 节点和连线使用独立 ID，支持单个元素级别的增量更新
- 版本号字段已预留（`version` + `last_edited_by`）

## 五、乐观并发控制（P1 即实现）

即使 P1 没有实时协作，多设备登录同一账号也可能产生冲突。

```typescript
// PUT /api/workflows/[id]/route.ts

export async function PUT(request: NextRequest, { params }: { params: { id: string } }) {
  const { userId } = await auth()
  const body = await request.json()

  // 乐观锁：客户端携带 version，服务端检查
  const result = await db
    .prepare(
      `
    UPDATE workflows
    SET data = ?, name = ?, updated_at = ?, version = version + 1, last_edited_by = ?
    WHERE id = ? AND user_id = ? AND version = ?
  `,
    )
    .bind(
      JSON.stringify(body.data),
      body.name,
      Date.now(),
      userId,
      params.id,
      userId,
      body.version, // 客户端传的版本号
    )
    .run()

  if (result.meta.changes === 0) {
    // 版本冲突
    return NextResponse.json(
      { error: 'Conflict', message: 'Workflow has been modified by another session' },
      { status: 409 },
    )
  }

  return NextResponse.json({ data: { version: body.version + 1 } })
}
```

## 六、P1 → P3 迁移路径

| 步骤 | 内容                                                     | 破坏性             |
| ---- | -------------------------------------------------------- | ------------------ |
| 1    | 创建 workspaces 和 workspace_members 表                  | 无（新增表）       |
| 2    | 为已有 workflows 的 workspace_id 设为 NULL（已是默认值） | 无                 |
| 3    | 实现 Durable Object 协作服务                             | 新增（不影响现有） |
| 4    | API 增加 workspace_id 过滤逻辑                           | 向后兼容（已预留） |
| 5    | 前端增加工作空间切换 UI                                  | 新增页面           |

**关键**：因为 P1 阶段已经预留了 `workspace_id`、`version`、`last_edited_by` 字段，P3 迁移时 **零破坏性**。
