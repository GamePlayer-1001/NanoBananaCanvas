# Agent 工作流画板完整方案

> 文档版本：v1.0  
> 创建日期：2026-04-29  
> 适用对象：Nano Banana Canvas 产品 / 设计 / 前端 / Agent 编排实现

---

## 一、目标定义

本方案用于把当前画板升级为“带 Agent 的工作流画板”。

核心目标：

1. 降低新用户上手门槛
2. 提高工作流创建效率
3. 让不会搭节点的人也能用起来
4. 让聊天成为工作流的高阶输入法
5. 让手工编辑成为 AI 编排后的精修层
6. 让画布始终是可验证、可回放、可修改的真相世界

一句话定义：

**用户先说目标，系统帮他搭工作流、润色提示词、返还画面提案并等待确认；确认后再执行，用户始终可以回到左侧画板继续编辑。**

---

## 二、产品定位

这不是“在画板右边加一个聊天框”。

这是一套新的产品工作方式：

1. 用户用自然语言表达目标
2. Agent 把目标翻译成结构化工作流
3. 画布承接最终结构与状态
4. 用户既可以继续对话，也可以直接接管画板

最终产品不是“聊天生图工具”，而是：

**一个由 Agent 协助编排、由画板承载真相的可视化工作流创作空间。**

---

## 三、核心设计原则

### 3.1 画布是真相源

所有最终结果必须落到现有工作流结构：

- `nodes`
- `edges`
- `node config`
- `execution history`

聊天不是第二套系统，只是更自然的输入方式。

### 3.2 聊天负责提案，画布负责落地

Agent 的职责不是黑盒替用户完成，而是输出：

- 要搭建什么工作流
- 要新增哪些节点
- 要修改哪些参数
- 为什么这么改
- 下一步建议是什么

### 3.3 先提案，再执行

尤其是涉及提示词润色、画面理解、大规模结构改动、高成本执行时：

1. 先理解用户意图
2. 先给出提案
3. 等用户确认
4. 再正式执行

### 3.4 原始意图必须保留

系统必须同时保留三层信息：

1. 用户原始需求
2. AI 理解后的画面提案
3. 最终执行提示词

AI 不得偷偷覆盖用户原始表达。

### 3.5 轻聊天，重反馈，少打扰

右侧聊天面板不应被分割成过多显式板块，不应把内部系统结构暴露给用户。

正确做法是：

- 面板外观简单
- 过程反馈自然
- 系统在聊天过程中用短句说明“自己正在做什么”
- 复杂结构放在系统内部，不污染聊天界面

## 四、交互总原则

### 4.1 右侧面板要简单，不要过度分区

不建议把右侧显式拆成：

- 目标输入区
- 操作区
- 解释区
- 状态区
- 建议区

这些结构可以存在于系统内部，但不直接暴露给用户。

用户看到的应该只是一个简单、可连续对话的面板。

### 4.2 系统要通过“过程消息”体现工作，而不是通过复杂 UI 教用户

正确体验示例：

用户：

`我想要一张男孩的图片，他穿着背带裤在打篮球`

系统过程消息：

1. `我先理解一下你想要的画面。`
2. `正在为你搭建一套基础生图工作流。`
3. `我已经补上了输入、提示词润色、图片生成和结果展示这几步。`
4. `工作流搭建完成，请于左侧画板中查看与编辑。`
5. `提示词润色已完成，我先把我理解的画面返给你确认。`

最终结束消息：

1. `工作流搭建完成，请于左侧画板中查看与编辑，需要我来详细为您介绍吗？`
2. `提示词已同步润色，您看看是否想要这个感觉。`
3. `画面提案：……`
4. `将用于生成的提示词：……`

### 4.3 系统要让用户觉得“安全”

右侧面板不应让用户觉得：

- 里面功能太多
- 不知道点哪里
- 怕操作错
- 怕把画布搞坏

所以聊天面板的主要职责不是展示全部能力，而是：

**用自然对话把复杂系统折叠起来，只在必要时把结果返给用户。**

## 五、核心用户路径

### 5.1 从空白开始

1. 用户说一句自然语言目标
2. Agent 理解目标
3. Agent 在左侧生成最小可运行工作流
4. Agent 润色提示词
5. Agent 将“画面提案 + 执行提示词”返给用户确认
6. 用户确认、修改或要求再来一版
7. 确认后执行
8. 生成后继续建议下一步

### 5.2 在已有工作流上继续改

1. 用户打开已有工作流
2. 用户说出增量需求
3. Agent 读取当前工作流摘要
4. Agent 产出结构化改动
5. 改动落到左侧画板
6. 系统用一句话说明本次改了什么

### 5.3 看不懂或跑不通时求助

1. 用户说“为什么这条链跑不通”
2. Agent 读取执行记录与节点摘要
3. Agent 定位问题
4. Agent 说明原因
5. Agent 提议直接修复或帮用户调整

---

## 六、六大核心能力

### 6.1 一句话生成基础工作流

这是最关键的 onboarding 能力。

示例：

- `帮我做一个电商主图生成流程`
- `做一个文案到配图到标题变体的流程`
- `做一个参考图到视频提示词再到视频生成的流程`

系统输出：

1. 直接在左侧生成可运行的最小工作流
2. 右侧用自然语言解释每个关键步骤
3. 给出下一步建议

价值：

- 用户不用从空白画布开始
- 用户第一次就能看到“系统真的在帮我做事”

### 6.2 对已有工作流做增量改造

这比从 0 创建更有实际价值。

示例：

- `在这个流程前面加一个风格分析步骤`
- `把这个图片节点换成更便宜的模型`
- `把文案生成拆成标题和正文两步`
- `把输出改成 4 个变体`

本质：

Agent 不是重建整张图，而是在现有工作流上产出结构化 patch。

### 6.3 工作流诊断师

用户示例：

- `为什么这条链跑不通？`
- `为什么这一步没有输出？`
- `哪里可以更省钱？`
- `这个流程太慢了，帮我优化一下`

系统能力：

1. 读取当前节点图
2. 读取执行记录
3. 读取任务状态
4. 解释原因
5. 提供修复建议

价值：

把“工作流编辑器”升级成“会指导用户的生产工具”。

### 6.4 结果反推下一步

生成完内容后，用户往往不知道怎么继续。

系统应主动建议：

- `要不要我基于这张图补一个视频生成分支？`
- `要不要再加一个标题文案分支？`
- `要不要为这张图批量生成 3 种风格变体？`

价值：

让工作流自然生长，而不是一次性结束。

### 6.5 模板化 + 对话化结合

流程示例：

1. 用户先选模板
2. Agent 用一句话解释模板
3. 用户说：`把这个模板改成适合服装商品的`
4. Agent 自动修改节点参数和结构

价值：

模板库不再是死模板，而是可对话改造的起点。

### 6.6 面向小白的目标式提问

右侧不应只是自由聊天，也应该提供轻量目标入口。

可内置为快捷建议：

- `我要做海报`
- `我要做视频`
- `我要分析一张图`
- `我要批量生成变体`
- `我要优化已有流程`

注意：

这些入口可以是轻按钮、建议气泡或输入框预设，不应把面板做成复杂后台。

## 七、提示词确认机制

### 7.1 为什么必须确认

用户最容易遇到的问题不是不会说，而是：

- 说得模糊
- 说得不完整
- AI 容易脑补过头

所以正确链路不是“用户一说 -> AI 直接出图”，而是：

**用户一说 -> AI 理解和润色 -> 返还画面提案 -> 用户确认 -> 再生成**

### 7.2 三层信息必须同时存在

系统应返给用户：

1. `你的原始想法`
2. `我理解的画面`
3. `将用于生成的提示词`

### 7.3 交互示例

用户：

`我想要一张男孩的图片，他穿着背带裤在打篮球`

系统：

`工作流已经帮你搭好了，请于左侧画板中查看与编辑。`

`我先把我理解的画面返给你确认：`

`画面提案：一个穿着牛仔背带裤的男孩，在室外篮球场运球或投篮，整体氛围有活力、少年感和运动感，阳光自然，动作抓拍明显。`

`将用于生成的提示词：A young boy wearing denim overalls playing basketball on an outdoor court...`

`你想要这个感觉吗？如果你愿意，我也可以帮你改得更写实、更像动漫，或者更像商业摄影。`

### 7.4 确认方式

用户可以：

1. 直接确认
2. 局部修改
3. 让系统再改一版
4. 选择不同风格方向
5. 自己编辑提示词

### 7.5 确认的产品原则

1. 默认先给一版合理方案
2. 不要无限追问
3. 复杂需求再多轮澄清
4. 先对齐画面，再消耗算力

## 八、标准工作流骨架

### 8.1 最小生图骨架

建议标准链路：

1. `User Intent`
2. `Prompt Refiner`
3. `Image Gen`
4. `Display`

### 8.2 为什么不直接把用户原话塞进生图节点

因为中间必须有“意图加工层”。

这层负责：

- 补细节
- 补结构
- 提升模型理解稳定性
- 保留用户原意

### 8.3 后续可扩展节点

未来可演进为：

1. `User Intent`
2. `Style Refiner`
3. `Prompt Refiner`
4. `Negative Prompt Builder`
5. `Image Gen`
6. `Display`

或者：

1. `User Intent`
2. `Prompt Variants`
3. `Variant Picker`
4. `Image Gen`
5. `Display`

## 九、Agent 的上下文理解机制

### 9.1 不要把整坨原始数据直接喂给 Agent

必须做一层“画布摘要器”。

### 9.2 画布摘要器至少输出这些内容

1. 当前有哪些节点
2. 每个节点类型与关键参数
3. 哪些节点已连接
4. 哪些节点没有输出
5. 最近一次执行失败在哪里
6. 当前选中的是谁
7. 当前工作流的主要目标是什么
8. 当前结果资产有哪些

### 9.3 价值

只有这样，AI 才像“看懂了画布”，而不是只是在猜。

## 十、产品辨识度体验

### 10.1 选中节点后进入节点语境

例如用户点中 `image-gen` 节点，右侧系统应自动知道：

- 这个节点当前在做什么
- 它吃什么输入
- 它产什么输出
- 用户现在可以对它说什么

这不一定要显式展示复杂结构，但系统回复应体现出这种上下文敏感性。

### 10.2 对话和画布双向同步

正确状态：

1. 用户手动拖了一个节点，右侧能理解
2. 用户在右侧说一句话，左侧能变化

这才是一体化，而不是两个孤岛。

### 10.3 每次变更都能被叙述

系统应主动反馈：

- `我刚帮你把单图生成改成了 4 图变体流程。`
- `我检测到你的输出没有接 Display，所以补了一个展示节点。`
- `我把这个文案节点拆成了标题和正文两步。`

这种反馈会增强信任感。

### 10.4 提供“从目标出发”的入口

你们当前更像“先搭积木，再做事”。

未来应变成：

**先说目标，再由系统帮你搭积木。**

这是产品范式变化的关键。

## 十一、右侧聊天面板的正确样子

### 11.1 面板外观原则

右侧应该看起来像一个简单助手，而不是一套复杂控制台。

不建议：

- 多 tab
- 多折叠区
- 显式工作流结构树
- 过细的功能分仓

建议：

- 一个主输入框
- 一段连续聊天历史
- 少量建议问题或快捷入口
- 过程反馈自然穿插在对话里

### 11.2 系统消息风格

系统消息应简短、自然、可信。

推荐语气：

- `我先帮你理解一下这个需求。`
- `正在搭建工作流。`
- `我已经补上了结果展示节点。`
- `工作流搭建完成，请于左侧画板中查看与编辑。`
- `提示词已经润色好了，您先看看我理解得对不对。`

### 11.3 面板的主要作用

右侧面板不负责展示全部系统结构，它负责：

1. 接收目标
2. 返回理解
3. 返回确认项
4. 解释改动
5. 给下一步建议

## 十二、技术落地原则

### 12.1 自然语言不能直接重写 workflow JSON

推荐链路：

`自然语言输入 -> 意图解析 -> 操作计划 -> workflow patch -> patch 应用 -> store 更新 -> 自动保存 -> 执行`

### 12.2 Agent 输出应是结构化操作

例如：

- `add_node`
- `remove_node`
- `update_node_config`
- `connect_nodes`
- `disconnect_edge`
- `duplicate_branch`
- `run_workflow`
- `request_prompt_confirmation`
- `explain_workflow`

### 12.3 大动作必须确认

以下情况建议先确认：

1. 删除大量节点
2. 大改提示词
3. 替换核心模型
4. 引入高成本执行
5. 重构主链路

### 12.4 小动作可自动落地

例如：

1. 补一个 Display 节点
2. 新增提示词润色节点
3. 小范围调参数
4. 新增建议分支但不自动运行

## 十三、版本路线

### V1：起步版

目标：

让完全不会搭节点的用户，也能一句话获得一条可运行工作流。

包含能力：

1. 一句话生成基础工作流
2. 提示词润色
3. 画面提案确认
4. 左侧画布同步生成
5. 基础工作流解释

### V2：改造版

目标：

让 Agent 真正能改造已有图，而不只是起步。

包含能力：

1. 增量修改已有工作流
2. 模板对话化改造
3. 成本/速度优化
4. 基础故障诊断
5. 结果反推下一步

### V3：共创版

目标：

让聊天与画布深度融合。

包含能力：

1. 节点语境自动理解
2. 更强的上下文摘要
3. 多版本提示词提案
4. 更长链路的创作记忆
5. 基于资产结果的自动分支建议

---

## 十四、成功标准

如果以下指标改善，说明方案方向正确：

1. 新用户首次成功运行工作流的时间下降
2. 空白画布放弃率下降
3. 用户手工创建基础节点的比例下降
4. 提示词确认后的首次命中率提升
5. 已有工作流被继续改造和复用的比例上升
6. 用户更愿意在左侧画板中继续精修

---

## 十五、结论

这套方案的本质，不是“让 AI 帮用户出图”，而是：

**让 AI 帮用户把模糊想法翻译成可运行、可理解、可确认、可修改的工作流。**

最终产品体验应是：

1. 用户从目标出发，而不是从节点出发
2. 系统用简单对话承接复杂能力，而不是把复杂度堆给用户
3. 左侧画板始终是唯一真相世界
4. 右侧 Agent 始终是创作协作者，而不是黑盒代工者

这才是“带 Agent 的工作流画板”最有价值、最有辨识度的形态。
