# Agent 工作流画板完整方案

> 文档版本：v1.2  
> 创建日期：2026-04-29  
> 更新日期：2026-04-30  
> 适用对象：Nano Banana Canvas 产品 / 设计 / 前端 / Agent 编排实现

---

## 一、目标定义

本方案用于把当前画板升级为“带 Agent 的工作流画板”。

核心目标：

1. 降低新用户上手门槛
2. 提高工作流创建效率
3. 让不会搭节点的人也能用起来
4. 让聊天成为工作流的高阶输入法
5. 让手工编辑成为 AI 编排后的精修层
6. 让画布始终是可验证、可回放、可修改的真相世界

一句话定义：

**用户先说目标，系统帮他搭工作流、润色提示词、返还画面提案并等待确认；确认后再执行，用户始终可以回到左侧画板继续编辑。**

---

## 二、产品定位

这不是“在画板右边加一个聊天框”。

这是一套新的产品工作方式：

1. 用户用自然语言表达目标
2. Agent 把目标翻译成结构化工作流
3. 画布承接最终结构与状态
4. 用户既可以继续对话，也可以直接接管画板

最终产品不是“聊天生图工具”，而是：

**一个由 Agent 协助编排、由画板承载真相的可视化工作流创作空间。**

---

## 三、核心设计原则

### 3.1 画布是真相源

所有最终结果必须落到现有工作流结构：

- `nodes`
- `edges`
- `node config`
- `execution history`

聊天不是第二套系统，只是更自然的输入方式。

### 3.2 聊天负责提案，画布负责落地

Agent 的职责不是黑盒替用户完成，而是输出：

- 要搭建什么工作流
- 要新增哪些节点
- 要修改哪些参数
- 为什么这么改
- 下一步建议是什么

### 3.3 该直接落图的直接落图，该确认的再确认

并不是所有请求都要先走“提案审批”。

尤其是创建工作流这类明确需求，应优先直接生成最小可运行工作流。

只有涉及提示词润色、画面理解、大规模结构改动、高成本执行时，才进入确认：

1. 先理解用户意图
2. 先判断这是“创建工作流”还是“需要确认的生成动作”
3. 创建工作流则先落图
4. 需要确认时再通过聊天返还画面提案与提示词
5. 等用户确认后再正式执行生成结果

### 3.4 原始意图必须保留

系统必须同时保留三层信息：

1. 用户原始需求
2. AI 理解后的画面提案
3. 最终执行提示词

AI 不得偷偷覆盖用户原始表达。

### 3.5 轻聊天，重反馈，少打扰

右侧聊天面板不应被分割成过多显式板块，不应把内部系统结构暴露给用户。

正确做法是：

- 面板外观简单
- 过程反馈自然
- 系统在聊天过程中用短句说明“自己正在做什么”
- 复杂结构放在系统内部，不污染聊天界面

## 四、交互总原则

### 4.1 右侧面板要简单，不要过度分区

不建议把右侧显式拆成：

- 目标输入区
- 操作区
- 解释区
- 状态区
- 建议区

这些结构可以存在于系统内部，但不直接暴露给用户。

用户看到的应该只是一个简单、可连续对话的面板。

### 4.1.1 面板形态应是右下角悬浮轻助手

正确形态不是固定右侧重侧栏，而是：

1. 默认停靠在右下角
2. 展开后浮在画布上层
3. 支持折叠 / 展开
4. 支持拖动到别处
5. 支持手动拖宽

### 4.2 系统要通过“过程消息”体现工作，而不是通过复杂 UI 教用户

正确体验示例：

用户：

`我想要一张男孩的图片，他穿着背带裤在打篮球`

系统过程消息：

1. `我先理解一下你想要的画面。`
2. `正在为你搭建一套基础生图工作流。`
3. `我已经补上了输入、提示词润色、图片生成和结果展示这几步。`
4. `工作流搭建完成，请于左侧画板中查看与编辑。`
5. `提示词润色已完成，我先把我理解的画面返给你确认。`

最终结束消息：

1. `工作流搭建完成，请于左侧画板中查看与编辑，需要我来详细为您介绍吗？`
2. `提示词已同步润色，您看看是否想要这个感觉。`
3. `画面提案：……`
4. `将用于生成的提示词：……`

### 4.3 系统要让用户觉得“安全”

右侧面板不应让用户觉得：

- 里面功能太多
- 不知道点哪里
- 怕操作错
- 怕把画布搞坏

所以聊天面板的主要职责不是展示全部能力，而是：

**用自然对话把复杂系统折叠起来，只在必要时把结果返给用户。**

## 五、核心用户路径

### 5.1 从空白开始

1. 用户说一句自然语言目标
2. Agent 理解目标
3. Agent 在左侧生成最小可运行工作流
4. Agent 润色提示词
5. Agent 将“画面提案 + 执行提示词”返给用户确认
6. 用户确认、修改或要求再来一版
7. 确认后执行
8. 生成后继续建议下一步

### 5.2 在已有工作流上继续改

1. 用户打开已有工作流
2. 用户说出增量需求
3. Agent 读取当前工作流摘要
4. Agent 产出结构化改动
5. 改动落到左侧画板
6. 系统用一句话说明本次改了什么

### 5.3 看不懂或跑不通时求助

1. 用户说“为什么这条链跑不通”
2. Agent 读取执行记录与节点摘要
3. Agent 定位问题
4. Agent 说明原因
5. Agent 提议直接修复或帮用户调整

## 六、六大核心能力

### 6.1 一句话生成基础工作流

这是最关键的 onboarding 能力。

示例：

- `帮我做一个电商主图生成流程`
- `做一个文案到配图到标题变体的流程`
- `做一个参考图到视频提示词再到视频生成的流程`

系统输出：

1. 直接在左侧生成可运行的最小工作流
2. 右侧用自然语言解释每个关键步骤
3. 给出下一步建议

价值：

- 用户不用从空白画布开始
- 用户第一次就能看到“系统真的在帮我做事”

### 6.2 对已有工作流做增量改造

这比从 0 创建更有实际价值。

示例：

- `在这个流程前面加一个风格分析步骤`
- `把这个图片节点换成更便宜的模型`
- `把文案生成拆成标题和正文两步`
- `把输出改成 4 个变体`

本质：

Agent 不是重建整张图，而是在现有工作流上产出结构化 patch。

### 6.3 工作流诊断师

用户示例：

- `为什么这条链跑不通？`
- `为什么这一步没有输出？`
- `哪里可以更省钱？`
- `这个流程太慢了，帮我优化一下`

系统能力：

1. 读取当前节点图
2. 读取执行记录
3. 读取任务状态
4. 解释原因
5. 提供修复建议

价值：

把“工作流编辑器”升级成“会指导用户的生产工具”。

### 6.4 结果反推下一步

生成完内容后，用户往往不知道怎么继续。

系统应主动建议：

- `要不要我基于这张图补一个视频生成分支？`
- `要不要再加一个标题文案分支？`
- `要不要为这张图批量生成 3 种风格变体？`

价值：

让工作流自然生长，而不是一次性结束。

### 6.5 模板化 + 对话化结合

流程示例：

1. 用户先选模板
2. Agent 用一句话解释模板
3. 用户说：`把这个模板改成适合服装商品的`
4. Agent 自动修改节点参数和结构

价值：

模板库不再是死模板，而是可对话改造的起点。

### 6.6 面向小白的目标式提问

右侧不应只是自由聊天，也应该提供轻量目标入口。

可内置为快捷建议：

- `我要做海报`
- `我要做视频`
- `我要分析一张图`
- `我要批量生成变体`
- `我要优化已有流程`

注意：

这些入口可以是轻按钮、建议气泡或输入框预设，不应把面板做成复杂后台。

## 七、提示词确认机制

### 7.1 为什么必须确认

用户最容易遇到的问题不是不会说，而是：

- 说得模糊
- 说得不完整
- AI 容易脑补过头

所以正确链路不是“用户一说 -> AI 直接出图”，而是：

**用户一说 -> AI 理解和润色 -> 返还画面提案 -> 用户修改确认 -> 再生成**

### 7.2 三层信息必须同时存在

系统应返给用户：

1. `你的原始想法`
2. `我理解的画面`
3. `将用于生成的提示词`

### 7.3 交互示例

用户：

`我想要一张男孩的图片，他穿着背带裤在打篮球`

系统：

`工作流已经帮你搭好了，请于左侧画板中查看与编辑。`

`我先把我理解的画面返给你确认：`

`画面提案：一个穿着牛仔背带裤的男孩，在室外篮球场运球或投篮，整体氛围有活力、少年感和运动感，阳光自然，动作抓拍明显。`

`将用于生成的提示词：A young boy wearing denim overalls playing basketball on an outdoor court...`

`你想要这个感觉吗？如果你愿意，我也可以帮你改得更写实、更像动漫，或者更像商业摄影。`

### 7.4 确认方式

用户可以：

1. 直接确认
2. 局部修改
3. 让系统再改一版
4. 选择不同风格方向
5. 自己编辑提示词

确认应优先通过聊天里的肯定性语言完成，而不是系统级审批按钮。

例如：

- `我确认`
- `可以执行`
- `是的，就这样`
- `继续`

### 7.5 确认的产品原则

1. 默认先给一版合理方案
2. 不要无限追问
3. 复杂需求再多轮澄清
4. 先对齐画面，再消耗算力

## 八、标准工作流骨架

### 8.1 最小生图骨架

建议标准链路：

1. `User Intent`
2. `Prompt Refiner`
3. `Image Gen`
4. `Display`

### 8.2 为什么不直接把用户原话塞进生图节点

因为中间必须有“意图加工层”。

这层负责：

- 补细节
- 补结构
- 提升模型理解稳定性
- 保留用户原意

### 8.3 后续可扩展节点

未来可演进为：

1. `User Intent`
2. `Style Refiner`
3. `Prompt Refiner`
4. `Negative Prompt Builder`
5. `Image Gen`
6. `Display`

或者：

1. `User Intent`
2. `Prompt Variants`
3. `Variant Picker`
4. `Image Gen`
5. `Display`

## 九、Agent 的上下文理解机制

### 9.1 不要把整坨原始数据直接喂给 Agent

必须做一层“画布摘要器”。

### 9.2 画布摘要器至少输出这些内容

1. 当前有哪些节点
2. 每个节点类型与关键参数
3. 哪些节点已连接
4. 哪些节点没有输出
5. 最近一次执行失败在哪里
6. 当前选中的是谁
7. 当前工作流的主要目标是什么
8. 当前结果资产有哪些

### 9.3 价值

只有这样，AI 才像“看懂了画布”，而不是只是在猜。

## 十、产品辨识度体验

### 10.1 选中节点后进入节点语境

例如用户点中 `image-gen` 节点，右侧系统应自动知道：

- 这个节点当前在做什么
- 它吃什么输入
- 它产什么输出
- 用户现在可以对它说什么

这不一定要显式展示复杂结构，但系统回复应体现出这种上下文敏感性。

### 10.2 对话和画布双向同步

正确状态：

1. 用户手动拖了一个节点，右侧能理解
2. 用户在右侧说一句话，左侧能变化

这才是一体化，而不是两个孤岛。

### 10.3 每次变更都能被叙述

系统应主动反馈：

- `我刚帮你把单图生成改成了 4 图变体流程。`
- `我检测到你的输出没有接 Display，所以补了一个展示节点。`
- `我把这个文案节点拆成了标题和正文两步。`

这种反馈会增强信任感。

### 10.4 提供“从目标出发”的入口

你们当前更像“先搭积木，再做事”。

未来应变成：

**先说目标，再由系统帮你搭积木。**

这是产品范式变化的关键。

## 十一、右侧聊天面板的正确样子

### 11.1 面板外观原则

右侧应该看起来像一个简单助手，而不是一套复杂控制台。

不建议：

- 多 tab
- 多折叠区
- 显式工作流结构树
- 过细的功能分仓

建议：

- 一个主输入框
- 一段连续聊天历史
- 少量建议问题或快捷入口
- 过程反馈自然穿插在对话里

建议首屏引导文案：

- `你今天想做些什么？`
- `我想生成一张小猫的图片`
- `我想将这个图片修改成写实风格`
- `为什么这条工作流跑不通`
- `向我介绍一下这条工作流`

### 11.2 系统消息风格

系统消息应简短、自然、可信。

推荐语气：

- `我先帮你理解一下这个需求。`
- `正在搭建工作流。`
- `我已经补上了结果展示节点。`
- `工作流搭建完成，请于左侧画板中查看与编辑。`
- `提示词已经润色好了，您先看看我理解得对不对。`

### 11.3 面板的主要作用

右侧面板不负责展示全部系统结构，它负责：

1. 接收目标
2. 返回理解
3. 返回确认项
4. 解释改动
5. 给下一步建议

## 十二、技术落地原则

### 12.1 自然语言不能直接重写 workflow JSON

推荐链路：

`自然语言输入 -> 意图解析 -> 操作计划 -> workflow patch -> patch 应用 -> store 更新 -> 自动保存 -> 执行`

### 12.2 Agent 输出应是结构化操作

例如：

- `add_node`
- `remove_node`
- `update_node_data`
- `connect`
- `disconnect`
- `focus_nodes`
- `run_workflow`
- `request_prompt_confirmation`
- `explain_workflow`

### 12.3 大动作必须确认

以下情况建议先确认：

1. 删除大量节点
2. 大改提示词
3. 替换核心模型
4. 引入高成本执行
5. 重构主链路

### 12.4 小动作可自动落地

例如：

1. 补一个 Display 节点
2. 新增提示词润色节点
3. 小范围调参数
4. 新增建议分支但不自动运行

## 十三、版本路线

### V1：起步版

目标：

让完全不会搭节点的用户，也能一句话获得一条可运行工作流。

包含能力：

1. 一句话生成基础工作流
2. 提示词润色
3. 画面提案确认
4. 左侧画布同步生成
5. 基础工作流解释

### V2：改造版

目标：

让 Agent 真正能改造已有图，而不只是起步。

包含能力：

1. 增量修改已有工作流
2. 模板对话化改造
3. 成本/速度优化
4. 基础故障诊断
5. 结果反推下一步

### V3：共创版

目标：

让聊天与画布深度融合。

包含能力：

1. 节点语境自动理解
2. 更强的上下文摘要
3. 多版本提示词提案
4. 更长链路的创作记忆
5. 基于资产结果的自动分支建议

---

## 十四、成功标准

如果以下指标改善，说明方案方向正确：

1. 新用户首次成功运行工作流的时间下降
2. 空白画布放弃率下降
3. 用户手工创建基础节点的比例下降
4. 提示词确认后的首次命中率提升
5. 已有工作流被继续改造和复用的比例上升
6. 用户更愿意在左侧画板中继续精修

---

## 十五、结论

这套方案的本质，不是“让 AI 帮用户出图”，而是：

**让 AI 帮用户把模糊想法翻译成可运行、可理解、可确认、可修改的工作流。**

最终产品体验应是：

1. 用户从目标出发，而不是从节点出发
2. 系统用简单对话承接复杂能力，而不是把复杂度堆给用户
3. 左侧画板始终是唯一真相世界
4. 右侧 Agent 始终是创作协作者，而不是黑盒代工者

这才是“带 Agent 的工作流画板”最有价值、最有辨识度的形态。

## 十六、视觉系统细化

### 16.1 整体气质

右侧 Agent 面板的视觉方向不应像客服聊天框，也不应像 IDE 调试台。

它应该更像：

1. 一个安静、可信、略带专业感的创作副驾驶
2. 和左侧白色画布形成“主工作区 + 辅助语境区”的关系
3. 有清晰层级，但不靠大面积色块制造压迫感

建议关键词：

- `clean`
- `calm`
- `structured`
- `editorial`
- `assistant-like but not robotic`

### 16.2 配色建议

遵循项目现有低饱和与非荧光原则，右侧面板建议使用中性底色 + 品牌蓝紫点缀，不要把整块面板做成深色。

建议 token：

```css
:root {
  --agent-panel-bg: #f7f8fc;
  --agent-panel-surface: rgba(255, 255, 255, 0.88);
  --agent-panel-surface-strong: #ffffff;
  --agent-panel-border: rgba(15, 23, 42, 0.08);
  --agent-panel-border-strong: rgba(15, 23, 42, 0.14);
  --agent-panel-text: #0f172a;
  --agent-panel-text-secondary: rgba(15, 23, 42, 0.68);
  --agent-panel-text-tertiary: rgba(15, 23, 42, 0.46);

  --agent-accent-hover: #4f46e5;

  --agent-warning: #d97706;

```

颜色职责建议：

1. `accent` 只用于主 CTA、选中态、正在执行态
2. `success` 用于“工作流已生成 / 已完成确认 / 已同步到画布”
3. `warning` 用于“建议确认 / 高成本执行 / 将替换模型”
4. `danger` 仅用于“删除节点 / 取消任务 / 回滚”
5. 大面积背景尽量使用灰白和半透明白，不要用高饱和品牌色铺底

### 16.3 层级关系

建议把右侧面板拆成 4 层视觉深度：

1. 最底层：轻微竖向渐变背景，承接整个 Agent 区域
2. 第二层：消息列表承载层，接近透明白卡
3. 第三层：消息气泡、状态条、提案卡片
4. 第四层：输入框、确认按钮、浮动建议

背景建议：

```css
background:
  radial-gradient(circle at top right, rgba(99, 102, 241, 0.08), transparent 34%),
  linear-gradient(180deg, #fbfbfe 0%, #f4f6fb 100%);
```

### 16.4 字体与信息密度

右侧面板不要延续 Landing 的强叙事字体，应回归可读性。

建议：

1. 标题与小标签使用当前系统 UI 字体
2. 系统过程消息使用 `13px-14px`
3. 核心提案正文使用 `14px-15px`
4. 提示词预览区域使用 `13px` + `font-mono` 混排或等宽局部字段
5. 每条消息控制在 2 到 5 行为主，避免一次塞入整屏说明

## 十七、组件搭配细化

### 17.1 右侧 Agent 面板推荐结构

建议采用单面板、弱分区方案：

1. `AgentPanel`
2. `AgentConversation`
3. `AgentProcessMessage`
4. `AgentProposalCard`
5. `AgentPromptCompareCard`
6. `AgentQuickActions`
7. `AgentComposer`

不是所有组件都长期显示，而是按状态组合。

### 17.2 面板骨架

```text
┌────────────────────────────────┐
│ Header                         │
│ 标题 / 当前模式 / 轻状态提示     │
├────────────────────────────────┤
│ Conversation Scroll Area       │
│ 用户消息                        │
│ 系统过程消息                    │
│ 提案卡片                        │
│ 确认卡片                        │
│ 诊断结论                        │
├────────────────────────────────┤
│ Quick Actions                  │
│ 建议气泡 / 风格捷径 / 常用操作   │
├────────────────────────────────┤
│ Composer                       │
│ 输入框 + 发送 + 附加动作        │
└────────────────────────────────┘
```

### 17.3 Header 设计

Header 不应该承担复杂控制，只承担上下文提示。

建议内容：

1. 当前模式：`开始搭建` / `修改当前工作流` / `诊断问题`
2. 当前上下文：`已连接到画板` / `已选中 Image Gen 节点`
3. 可选小按钮：`查看改动`、`回到最新提案`

不建议放：

1. 多标签页
2. 复杂筛选
3. 显式系统配置
4. 大片帮助文档入口

### 17.4 消息类型

右侧消息建议至少区分 6 种：

1. `user-message`
2. `assistant-message`
3. `process-message`
4. `proposal-message`
5. `confirmation-message`
6. `diagnosis-message`

视觉差异建议：

1. 用户消息更实体，轻浅色底
2. 助手普通消息透明度更高，更轻
3. 过程消息使用无气泡文本 + 小图标 + 极轻 loading 线
4. 提案消息使用卡片
5. 确认消息强调主操作按钮
6. 诊断消息增加状态色侧边条

### 17.5 Proposal Card 组件

这是整个方案里最值得精修的组件。

内容建议：

1. `我准备怎么改`
2. `会新增哪些节点`
3. `会改哪些参数`
4. `为什么这么改`
5. `是否需要你确认`

推荐字段：

```ts
type WorkflowProposalCard = {
  title: string
  summary: string
  changes: Array<{
    kind: 'add_node' | 'update_node' | 'connect' | 'remove_node' | 'run'
    label: string
    detail: string
    risk: 'low' | 'medium' | 'high'
  }>
  rationale?: string
  requiresConfirmation: boolean
}
```

### 17.6 Prompt Compare Card 组件

提示词确认不要只给一大段文本，建议使用三段结构：

1. 原始意图
2. 画面提案
3. 执行提示词

推荐交互：

1. 默认展示提炼版
2. 支持展开查看完整 prompt
3. 支持一键“更写实 / 更商业 / 更动漫 / 更克制”
4. 支持“直接用于执行”

### 17.7 Quick Actions 组件

Quick Actions 应该是轻量浮动建议，而不是功能菜单。

推荐文案：

1. `改成更写实`
2. `加一个风格分析节点`
3. `生成 4 个变体`
4. `解释这条链在做什么`
5. `帮我优化成本`

### 17.8 Composer 输入区

输入区建议具备这些能力：

1. 单一主输入框
2. Enter 发送，Shift+Enter 换行
3. 支持引用当前选中节点语境
4. 支持展示“即将影响当前工作流”的轻提示

不建议第一版做：

1. 多模态附件堆叠上传器
2. 复杂 slash commands
3. 太多输入模式切换

## 十八、交互动画细化

### 18.1 动效原则

Agent 面板动效必须满足四个原则：

1. 反馈比表演重要
2. 状态变化必须被看见
3. 批量动画不能拖慢输入节奏
4. 不得和画布拖拽产生抢焦点

### 18.2 推荐动效节奏

建议统一 motion token：

```ts
const motion = {
  fast: 0.18,
  base: 0.24,
  slow: 0.36,
  easeOut: [0.22, 1, 0.36, 1],
  easeSoft: [0.25, 0.46, 0.45, 0.94],
}
```

### 18.3 动效分配

1. 面板首次进入：`fade + x: 12 -> 0`，时长 220ms
2. 新消息进入：`fade + y: 8 -> 0`，时长 180ms
3. 过程消息更新：文本淡切，不做跳动
4. Proposal Card 出现：`scale 0.98 -> 1 + fade`
5. Prompt Compare Card 展开：高度过渡 + 内容 stagger
6. Quick Action chip hover：背景轻推亮，不做弹跳
7. 确认按钮：hover 位移 `translateY(-1px)`，active 回落

### 18.4 画布联动动效

聊天触发画布变化时，左侧应有“被改动”的可见反馈。

建议：

1. 新增节点时轻微淡入并带 `scale 0.96 -> 1`
2. 新连线时 path 描边从 0 到 100%
3. 被修改节点加 `2s` 的柔和高亮环
4. 面板中点击“查看改动”时自动 `fitView` 到新增节点簇

### 18.5 加载态设计

不要用笨重 spinner 占据聊天区。

建议拆成三类：

1. 理解中：三点脉冲或细横条 shimmer
2. 搭建中：过程消息逐条出现
3. 等确认：静态卡片 + 明确 CTA

### 18.6 降级策略

必须支持：

1. `prefers-reduced-motion`
2. 低性能设备关闭背景渐变位移
3. 画布交互期间减少右侧非必要动画

## 十九、实现细节细化

### 19.1 核心状态机

Agent 面板不要只靠聊天消息数组驱动，至少需要一层会话状态机。

建议状态：

1. `idle`
2. `understanding`
3. `planning`
4. `patch-ready`
5. `awaiting-confirmation`
6. `applying-patch`
7. `ready-to-run`
8. `running`
9. `diagnosing`
10. `error`

价值：

1. UI 不会乱
2. 不同卡片显示条件清晰
3. 可以明确哪些动作允许自动执行，哪些必须确认

### 19.2 自然语言到工作流的链路

推荐固定链路：

`user message -> intent parser -> context summarizer -> planner -> operation plan -> workflow patch -> patch apply -> save -> optional run`

不要跳成：

`user message -> LLM 直接改整份 workflow json`

### 19.3 Context Summarizer 输出

推荐输出结构：

```ts
type CanvasSummary = {
  workflowId: string
  nodeCount: number
  edgeCount: number
  selectedNodeId?: string
  selectedNodeType?: string
  nodes: Array<{
    id: string
    type: string
    label: string
    status?: string
    inputs?: string[]
    outputs?: string[]
    keyConfig?: Record<string, unknown>
  }>
  issues: Array<{
    code: string
    message: string
    nodeId?: string
  }>
  latestExecution?: {
    status: 'idle' | 'running' | 'completed' | 'failed'
    failedNodeId?: string
    failedReason?: string
  }
}
```

### 19.4 Planner 输出

Planner 不应直接产出自由文本，而应该产出两层结果：

1. 面向系统的结构化 patch plan
2. 面向用户的自然语言说明

推荐结构：

```ts
type AgentPlan = {
  goal: string
  mode: 'create' | 'update' | 'diagnose' | 'optimize'
  requiresConfirmation: boolean
  reasons: string[]
  operations: WorkflowOperation[]
  userFacingSummary: string
}
```

### 19.5 Workflow Operation 协议

建议第一版操作集控制在有限范围：

```ts
type WorkflowOperation =
  | { type: 'add_node'; nodeType: string; position?: { x: number; y: number }; initialData?: Record<string, unknown> }
  | { type: 'update_node_data'; nodeId: string; patch: Record<string, unknown> }
  | { type: 'remove_node'; nodeId: string }
  | { type: 'connect'; source: string; sourceHandle?: string; target: string; targetHandle?: string }
  | { type: 'disconnect'; edgeId: string }
  | { type: 'focus_nodes'; nodeIds: string[] }
  | { type: 'request_prompt_confirmation'; payload: PromptConfirmationPayload }
  | { type: 'run_workflow'; scope?: 'all' | 'from-node'; nodeId?: string }
```

这样做的好处：

1. 和 `useFlowStore` 现有能力天然对接
2. 能走审计日志
3. 能做撤销
4. 能限制 Agent 权限边界

### 19.6 Patch Apply 层

建议新增一个中间层，而不是让聊天面板直接改 store：

1. `buildOperationPlan()`
2. `validateOperationPlan()`
3. `applyOperationPlanToStore()`
4. `persistWorkflowDraft()`

原因：

1. 聊天 UI 和画布编辑逻辑解耦
2. 以后同样能支持模板导入和批量改造
3. 可以集中做权限与风险检查

### 19.7 自动确认与强制确认规则

建议明确规则：

自动落地：

1. 补 Display
2. 增加 Prompt Refiner
3. 轻微参数修改
4. 给节点补默认值

强制确认：

1. 删除节点
2. 替换核心模型
3. 改动超过 4 个节点
4. 执行高成本图片/视频任务
5. 覆盖用户已编辑的 prompt 主文本

### 19.8 聊天消息与任务系统的关系

当前项目已有 `lib/tasks/service.ts` 的异步任务底层，因此 Agent 面板要避免自己重新发明任务系统。

推荐做法：

1. Agent 负责“计划、改图、确认、发起执行”
2. 真正执行仍走现有 `run_workflow` 或异步 `tasks` 主链
3. 聊天区只订阅任务状态，不自己持有另一套执行真相源

### 19.9 失败恢复

Agent 修改失败时，不要只报一句失败。

建议返回：

1. 哪一步失败了
2. 是否已经部分修改画布
3. 是否已自动回滚
4. 用户下一步可以做什么

### 20.1 前端目录建议

如果正式落地，推荐新增独立 Agent 模块，而不是把逻辑全塞进 `components/canvas/canvas.tsx`。

```text
  agent-quick-actions.tsx

apps/web/stores/
  use-agent-store.ts

apps/web/lib/agent/
  types.ts
  constants.ts

  build-agent-plan.ts
  validate-agent-plan.ts
  apply-agent-plan.ts
  explain-agent-change.ts

apps/web/hooks/
  use-agent-session.ts
  use-agent-actions.ts
```

### 20.2 职责划分

建议分层：

1. `components/agent/*`
   只负责 UI
2. `stores/use-agent-store.ts`
   负责会话状态与视图态
3. `lib/agent/summarize-canvas.ts`
   负责把 ReactFlow 世界压缩成 Agent 可读摘要
4. `lib/agent/build-agent-plan.ts`
   负责调用 LLM / Planner
5. `lib/agent/apply-agent-plan.ts`
   负责把结构化操作映射到 `useFlowStore`

### 20.3 与现有 store 的衔接

当前已有：

1. `useFlowStore`
2. `useExecutionStore`
3. `useHistoryStore`

建议新增 `useAgentStore`，但不要重复持有以下真相：

1. `nodes`
2. `edges`
3. `viewport`
4. `execution results`

`useAgentStore` 只应该存：

1. 会话消息
2. 当前模式
3. 待确认提案
4. 当前选中节点语境
5. 本轮操作状态

### 20.4 API 边界建议

建议新增单独 API 语义域：

1. `POST /api/agent/plan`
2. `POST /api/agent/explain`
3. `POST /api/agent/diagnose`
4. `POST /api/agent/refine-prompt`

第一版不要直接做一个万能 `POST /api/agent/chat`，否则后续会变成巨石接口。

### 20.5 数据审计

每次 Agent 改动画布，建议记录：

1. 用户原话
2. 系统摘要
3. 结构化操作
4. 是否需要确认
5. 最终是否执行

这会非常有价值：

1. 用于回放
2. 用于调试 Agent 错误
3. 用于后续产品分析

## 二十一、规避点与反模式

### 21.1 绝对不要做的事

1. 让 LLM 直接输出整份 workflow JSON 并覆盖本地状态
2. 让聊天区拥有独立于画布之外的第二套节点真相源
3. 为了“聪明”而自动删除用户节点
4. 把所有 Agent 能力一次性堆成复杂控制台
5. 用过重动效拖慢输入、拖拽和执行反馈

### 21.2 高风险误区

误区一：把右侧做成很多 tab  
结果：用户不知道该去哪一步，心智分裂

误区二：让 Agent 一上来连续追问  
结果：上手成本反而升高

误区三：每次改动都全量重建工作流  
结果：用户失去信任，历史编辑被破坏

误区四：诊断时只看报错文本，不看节点上下文和执行记录  
结果：诊断看起来很像 AI 在胡猜

### 21.3 体验规避点

1. 过程消息不要太长
2. 一次提案不要超过 5 个主要动作
3. prompt 确认不要塞成难读的大段英文
4. 修改完成后一定要告诉用户“左侧哪里变了”
5. 对高成本行为必须显式提醒

### 21.4 工程规避点

1. 不要把 Agent 逻辑耦合进 `Canvas` 主组件
2. 不要在组件里直接拼 operation plan
3. 不要把确认态只存在本地临时变量里
4. 不要绕过 `useHistoryStore`，否则回滚体验会断
5. 不要新增与现有 `tasks` 主链冲突的执行状态字段

### 21.5 第一版实现边界建议

第一版建议只做：

1. 一句话生成最小工作流
2. 提示词润色与确认
3. 对已有工作流做小范围增量修改
4. 对失败工作流做基础诊断

第一版先不要做：

1. 多 Agent 并行讨论
2. 长记忆创作档案
3. 自动大规模重构
4. 复杂多模态聊天上传
5. 太强的自主执行

### 21.6 成功的判断标准

如果细化后的实现是正确的，用户应该感受到：

1. 我不需要先学节点，也能开始做事
2. 系统会先理解我，再替我动画布
3. 它每次改动都说得清楚
4. 我始终能在左侧把控最终结构
5. 我没有被一个“聪明但不受控的黑盒”劫持
